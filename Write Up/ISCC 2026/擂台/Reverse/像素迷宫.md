
## 一、题目剖析

题目的名称为“像素迷宫”，附带的描述语为：

> 是像素不是 0 和 1。

这句提示至关重要。它向我们传达了一个信息：尽管题目名称带有“像素”二字，我们却不能简单地将其视作普通的黑白二值图像。换言之，不能仅仅陷入 `0` 代表障碍物、`1` 代表通道的固定思维中。

通常在遇到这种类型的逆向工程题目时，解题核心在于：程序底层通常隐匿着一个“迷宫地图”，而我们需要输入的 flag 实质上是一条行进路径。程序会依据这串字符来校验我们能否从起始位置顺利抵达终点。

该题目最终要求提交的格式如下：

```text
ISCC{路径字符串}
```

其中，代表路径的字符由 `E`、`N`、`S`、`W` 构成，分别对应着东、北、南、西这四个基本方位。

单凭这些命名，我们即可对其功能进行初步的猜想：

| 函数名称            | 预期功能                                     |

以上过程是十分关键的。开展逆向分析切忌毫无头绪地去阅读海量的反汇编代码，首要任务应当是精准定位那些与输入校验紧密结合的关键模块。

## 五、剖析 main 入口点

启动 IDA 或是 Ghidra 来加载该程序，并跳转至 `main` 函数处。

此主函数的控制流相当清晰，将其逆向还原后的伪代码大体如下：

```cpp
int main() {
```

由此可见，程序的重头戏并未集中在 `main` 内部，而是交由以下两个子程序去处理：

```cpp
extractPath(input)
```

即遵循着“提取行进路线 -> 校验路线正确性”的先后流程。

## 六、探究 extractPath：明确 flag 构造特征

接着我们审视 `extractPath` 函数。

该模块主要承担了三个方面的检验工作。

### 1. 验证字符串长度

在汇编代码层面上存在如下形式的判定指令：

```asm
cmp QWORD PTR [rdx+0x8], 0x6
jbe ...
```

这表明用户提供的输入字符总数必须大于 6。

原因何在呢？

这主要是因为即便里面没有任何内容，最基本的结构也应该是这样：

```text
ISCC{}
```

其本身占据了 6 个字符的长度。若总长不足此数值，那么它显然不是一个符合规范的 flag。

------

### 2. 校验头部标志是否匹配 ISCC{

在分析 `extractPath` 时能够留意到，程序会截取输入数据的前 5 位，接着将其同下列字符串实施比对：

```text
ISCC{
```

所以，这就要求提交的数据必须要由以下字串：

```text
ISCC{
```

作为起始部分。

------

### 3. 判断末尾字符是否是 }

在反汇编界面里我们能发现另一条比对指令：

```asm
cmp BYTE PTR [rax+r9*1-0x1], 0x7d
```

其中，`0x7d` 正好是下面这个 ASCII 字符的编码值：

```text
}
```

故而要求用户键入的最后一位绝对要是闭合的花括号。

------

### 4. 审核内部的路径标识字符

紧接着，程序会把被包裹在括号内的有效内容提取出来，也就是对应着：

```text
ISCC{这里的内容}
```

括号以内的字符组合必须完全由下面这几种字母构成：

```text
E N S W
```

很容易就能联想到，这里的 `E`、`N`、`S`、`W` 代表着移动方向：

| 字母 | 具体释义           |
| ---- | ------------------ |
| `E`  | East，代表往右侧移动 |
| `W`  | West，代表往左侧移动 |
| `S`  | South，代表往下方移动 |
| `N`  | North，代表往上方移动 |

借由上述分析，可以归纳出 flag 的整体全貌：

```text
flag = ISCC{一串由 E/N/S/W 组成的路径}
```

## 七、拆解 validatePath：明晰起始、目标位置及游走法则

我们转去分析 `validatePath` 函数。

该过程会逐一读取路线里的每一项字符指令。在其局部空间中声明了两个用于记录坐标的变量，大致等同于：

```cpp
int row = 0;
int col = 0;
```

这便揭示了迷宫的起始坐标点位于：

```text
(0, 0)
```

随后便会针对路线里的所有步骤展开逐一核查。

其核心校验逻辑大致可以表述如下：

```cpp
bool validatePath(const std::string& path) {
    int row = 0;
    int col = 0;

    for (int i = 0; i < path.length(); i++) {
        char dir = path[i];

        if (!isValidMove(row, col, dir)) {
            return false;
        }

        if (dir == 'N') row--;
        if (dir == 'S') row++;
        if (dir == 'E') col++;
        if (dir == 'W') col--;

        if (row < 0 || row > 9 || col < 0 || col > 9) {
            return false;
        }

        if (row == 9 && col == 9) {
            return i == path.length() - 1;
        }
    }

    return false;
}
```

从这部分代码中，我们能够解读出数条重要的线索：

首先，该迷宫的矩阵规模为 `10 × 10`。

这是因为行列坐标的边界限定在了如下区间内：

```text
0 <= row <= 9
0 <= col <= 9
```

其次，出发点处在整个矩阵的左上端位置：

```text
(0, 0)
```

再次，目标终点设在矩阵的右下方：

```text
(9, 9)
```

最后，玩家给出的路线序列必须在执行完最后一个字符时，不偏不倚恰好落在终点。

假若在中间某一步就已经抵达目标位置，但指令流仍有残余部分，系统依然会判定此次行走失败。

## 八、探索 decryptPixel：揭晓像素还原策略

接着把目光投向 `decryptPixel` 逻辑。

这里的底层汇编代码显得极其短小精悍：

```asm
mov eax, ecx
xor eax, 0x7f
ret
```

显而易见，用于还原真实像素数值的算法公式表现为：

```cpp
decrypted_pixel = encrypted_pixel ^ 0x7f;
```

如果将其用 Python 语句来表达，则为：

```python
pixel = encrypted_pixel ^ 0x7f
```

这意味着在程序内部构建迷宫的像素数据并没有以裸数据（明文）形态存放，而是提前经历了一次按位异或混淆操作。

混淆所使用的密钥常量正是：

```text
0x7f
```

## 九、深究 getDecryptedPixel：定位迷宫矩阵位置

`getDecryptedPixel(row, col)` 的职责在于，凭借给定的行列参数提取出预存的加密像素，随后启动异或运算予以解密。

在该函数的代码块中，能够观察到数据被切分成了三个块进行调用：

```cpp
if (row <= 2) {
    return ENCRYPTED_PART1[row * 10 + col] ^ 0x7f;
} else if (row <= 6) {
    return ENCRYPTED_PART2[(row - 3) * 10 + col] ^ 0x7f;
} else {
    return ENCRYPTED_PART3[(row - 7) * 10 + col] ^ 0x7f;
}
```

上述逻辑暴露出，这张全尺寸为 `10 × 10` 的地图被肢解为了三个区块分别管理：

| 区块名称            | 映射的行数范围            |
| ------------------- | ------------------------- |
| `ENCRYPTED_PART1`   | 从索引 0 至索引 2 的行，合计 3 行 |
| `ENCRYPTED_PART2`   | 从索引 3 至索引 6 的行，合计 4 行 |
| `ENCRYPTED_PART3`   | 从索引 7 至索引 9 的行，合计 3 行 |

计算其总和：

```text
3 × 10 + 4 × 10 + 3 × 10 = 100
```

恰好与整个 `10 × 10` 迷宫所含的 100 个点位相吻合。

## 十、通过二进制文件扒取地图数据

### 1. 梳理 .rdata 节区参数

执行以下命令：

```bash
objdump -h PixelMaze.exe
```

我们能够观察到关于 `.rdata` 节区的属性大致如下：

```text
Idx Name    Size      VMA               File off
2   .rdata  00010220  00000001400bd000  000bb800
```

此输出表明了 `.rdata` 节的内存虚拟基地址设定为：

```text
0x1400bd000
```

而其对应的物理文件偏移起点为：

```text
0xbb800
```

假设我们发现某段全局数据在内存态中的虚拟地址为 `0x1400bd180`，那么就能换算出其在二进制文件内的真实偏移量为：

```text
0xbb800 + (0x1400bd180 - 0x1400bd000)
= 0xbb800 + 0x180
= 0xbb980
```

------

### 2. 获取第一部分与第二部分的密文

在查阅 `getDecryptedPixel` 的底层汇编时能够捕捉到类似如下的取址操作：

```asm
lea rax,[rip+...]  # 1400bd180 <ENCRYPTED_PART1>
lea rax,[rip+...]  # 1400bd0e0 <ENCRYPTED_PART2>
```

由此可得：

```text
ENCRYPTED_PART1 的内存位置：0x1400bd180
ENCRYPTED_PART2 的内存位置：0x1400bd0e0
```

将其转换为文件中的静态偏移值则是：

```text
PART1 静态文件偏移 = 0xbb800 + 0x180
PART2 静态文件偏移 = 0xbb800 + 0x0e0
```

由于数组内部的单项均为占用 4 bytes 的整数，即 DWORD 类型。

因此可以断定：

`PART1` 包含着 30 枚这类整数值。

`PART2` 包含着 40 枚这类整数值。

------

### 3. 解析略显棘手的 PART3 数据

和前两段截然不同，`PART3` 并不是静态地栖息于 `.rdata` 节内，而是借由 `getDecryptedPixel` 顶部的多条 `movabs` 汇编指令，被动态地拼凑到栈区中。

举个例子：

```asm
movabs rax, 0xbe000000be
mov QWORD PTR [rsp], rax
```

这段代码的目的是向栈中灌入 8 bytes 大小的数据。

鉴于 x86 架构遵循小端字节序（Little-Endian），上述数值在内存中展现出来的样貌其实是：

```text
be 00 00 00 be 00 00 00
```

假如我们将这 8 字节视为并排的两个 32 bit 整数，便能拆分为：

```text
0xbe, 0xbe
```

据此，我们只要仔细梳理所有的 `movabs` 指令，便能以人工的方式拼装出 `PART3` 的完整内容。

针对刚入门的分析者，有两种策略可供选择：

方案 A：借助 Ghidra 等工具的伪代码视图，直接浏览所呈现出的局部变量数组，这种方式最为清晰明了。

方案 B：通过审查汇编代码段内的 `movabs` 传值动作，将赋给寄存器的 8 字节值一分为二，提取出相关的 32 位整数。

## 十一、解读 isValidMove：领悟“并非简单的 0 与 1”

此处乃是解答本道题目的灵魂所在。

函数 `isValidMove(row, col, dir)` 首先会获取所处坐标的明文像素：

```cpp
pixel = getDecryptedPixel(row, col);
```

紧随其后，会根据传入的方向标志位去测试该 pixel 变量上的特定比特位。

将汇编代码转换为高级语义后呈现为：

```cpp
if (dir == 'E') return pixel & 1;
if (dir == 'W') return (pixel >> 1) & 1;
if (dir == 'S') return (pixel >> 2) & 1;
if (dir == 'N') return (pixel >> 3) & 1;
```

这一机制清晰地宣告：迷宫中的像素绝非局限于代表通与不通的 `0` 或 `1`。

实际上，每一个像素值充当着一个涵盖方位信息的位掩码（Bitmask）。

其占据的低四位比特（bit）各自映射着朝不同方位穿行的许可权：

| 比特位 | 对应方位 | 实际意义                 |
| ------ | -------- | ------------------------ |
| bit0   | `E`      | 是否准许往右侧移动       |
| bit1   | `W`      | 是否准许往左侧移动       |
| bit2   | `S`      | 是否准许往下方移动       |
| bit3   | `N`      | 是否准许往上方移动       |

打个比方：

假设迷宫某一单元格经过解密得到的像素二进制值为：

```text
0b0101
```

换算成常用的十进制即为 `5`。

不难看出，它最低位（bit0）以及第三低位（bit2）的数值均为 1，故而代表着：

```text
当前所在的这个单元格既允许向 E 方位行进，也允许向 S 方位迈步。
```

这恰恰呼应了题设提示中“是像素不是 0 和 1”所潜藏的深刻内涵。

这意味着我们面对的并非传统的二值迷宫地图，相反，程序巧妙地将通达各个方向的权限压缩进每一个像素自身的二进制比特位当中。

## 十二、将迷宫机制转换为图遍历模型

总结目前的发现：

起始坐标点位于：

```text
(0, 0)
```

目标通关点设定在：

```text
(9, 9)
```

整体的迷宫长宽尺度被划定为：

```text
10 × 10
```

地图内各个单元格的有效像素值需要通过如下算法：

```python
pixel = encrypted_pixel ^ 0x7f
```

来进行还原。

获取到明文数值后，又要依托其低位的四个比特来裁定周遭四个方位是否能够通行。

至此，本道题目的重心已然由最初的逆向工程剖析，平稳过渡到了一道图论搜索题。

我们可以借助广度优先搜索算法（BFS），沿着从 `(0,0)` 至 `(9,9)` 的路线进行探测。

采用 BFS 方案的优势在于算法思维直截了当，再者，假如存在多条成功通往终点的方式，BFS 可以确保我们率先探寻到步数相对更少的路线。

## 十三、编写通关脚本

接下来的代码是一份用于演示目的的求解程序。

```python
import struct
from collections import deque

filename = "PixelMaze.exe"

with open(filename, "rb") as f:
    data = f.read()

# .rdata 的文件偏移
rdata_file_offset = 0xbb800

# ENCRYPTED_PART2 位于 .rdata + 0xe0，共 40 个 DWORD
part2_offset = rdata_file_offset + 0xe0
part2 = list(struct.unpack("<40I", data[part2_offset:part2_offset + 40 * 4]))

# ENCRYPTED_PART1 位于 .rdata + 0x180，共 30 个 DWORD
part1_offset = rdata_file_offset + 0x180
part1 = list(struct.unpack("<30I", data[part1_offset:part1_offset + 30 * 4]))

# ENCRYPTED_PART3 是 getDecryptedPixel 中在栈上构造的 30 个 DWORD
part3 = [
    0xbe, 0xbe, 0x37, 0x2e, 0x7e, 0xb7, 0x8b, 0x1d, 0x66, 0x9d,
    0x6e, 0x4a, 0x3c, 0x5a, 0x7d, 0xa9, 0x3e, 0xee, 0x2e, 0xeb,
    0x76, 0xee, 0x4d, 0x0c, 0x67, 0x49, 0xad, 0x7c, 0x3d, 0x00
]

# 拼成完整的 10 × 10 加密迷宫
encrypted_maze = part1 + part2 + part3

assert len(encrypted_maze) == 100

# 解密
maze = [x ^ 0x7f for x in encrypted_maze]

# 每个方向对应的 bit 位
direction_bit = {
    "E": 0,
    "W": 1,
    "S": 2,
    "N": 3,
}

# 每个方向对应的坐标变化
direction_delta = {
    "E": (0, 1),
    "W": (0, -1),
    "S": (1, 0),
    "N": (-1, 0),
}

def can_move(row, col, direction):
    pixel = maze[row * 10 + col]
    bit = direction_bit[direction]
    return ((pixel >> bit) & 1) == 1

def bfs():
    start = (0, 0)
    target = (9, 9)

    q = deque()
    q.append((0, 0, ""))

    visited = set()
    visited.add(start)

    while q:
        row, col, path = q.popleft()

        if (row, col) == target:
            return path

        for direction in "NESW":
            if not can_move(row, col, direction):
                continue

            dr, dc = direction_delta[direction]
            nr = row + dr
            nc = col + dc

            if not (0 <= nr <= 9 and 0 <= nc <= 9):
                continue

            if (nr, nc) in visited:
                continue

            visited.add((nr, nc))
            q.append((nr, nc, path + direction))

    return None

path = bfs()

print("path =", path)
print("flag =", "ISCC{" + path + "}")
```

上文代码中的枚举寻路语句书写为：

```python
for direction in "NESW":
```
