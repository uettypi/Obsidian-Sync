
## 题目分析

提取并分析 APK 附件的整体架构后，能够迅速锁定以下几个核心要素：

- 包含特定的自定义 Java 类：`FlagValidator`、`CryptoEngine`、`Telemetry`、`NativeBridge`
- 引入了原生动态链接库：`libdeepseal.so`
- 在 `assets` 资源目录中发现了一个极其关键的数据文件：`telemetry.bin`

由此推测本题的整体检验机制应为：

1. 首先在 Java 层对用户输入执行初步的数据混淆与转换；
2. 随后将处理结果传递至 native 层展开二次验证；
3. 上述 `telemetry.bin` 文件则负责存放加密参数及最终用于比对的密文数据。

## Java 层逻辑

### 1. FlagValidator.check

`FlagValidator.check` 方法的执行流程相对清晰：

- 首先检验用户输入字符串的格式，确认其是否包含 `ISCC{` 前缀以及 `}` 后缀；
- 接着截取花括号内部的核心内容；
- 随后将该内容作为参数传入 `CryptoEngine.transform` 进行处理；
- 最终将该处理的返回值移交给 `NativeBridge.verify` 做进一步鉴定。

其核心代码逻辑大致等同于如下实现：

```java
public static boolean check(Context ctx, String s) {
    if (s.startsWith("ISCC{") && s.endsWith("}")) {
        String body = s.substring(5, s.length() - 1);
        return NativeBridge.verify(ctx, CryptoEngine.transform(ctx, body));
    }
    return false;
}
```

### 2. Telemetry.load

`Telemetry.load` 负责解析 `assets/telemetry.bin` 资源，并从中提取出四行关键数据：

```text
0b04110916060d
1337c0de42a5197f
Qh2s7Ca9VxN4fKpB8J1mL+zoF0ewAyOu3DrRYSgPItU6McdkXl/WTnGEZ5ivbHjq
T8GHtJzYDRX7siXzWKLJCGlrpi1sVwnvMhZCPj3jwDWYU4Y=
```

具体对应关系如下：

- 第 1 行对应 `shifts` 数组；
- 第 2 行对应 `mask` 密钥；
- 第 3 行为非标准的 Base64 编码表；
- 第 4 行则是最终需要进行匹配的目标密文。

### 3. CryptoEngine.transform

结合反编译工具的输出结果，我们可以将 Java 层的加密算法重构为如下代码：

```python
prev = 0
for i, c in enumerate(flag_body_ascii):
    v = c
    if 32 <= v <= 126:
        v = ((v - 32 + shifts[i % len(shifts)]) % 95) + 32
    prev = ((v ^ mask[i % len(mask)]) + prev + 3 * i) & 0xff
    out[i] = prev
```

这段代码的运算规则如下：

- 针对可打印的 ASCII 字符执行循环移位操作；
- 将其与 `mask` 数组中的元素进行循环异或运算；
- 随后将上一轮计算得出的状态值 `prev` 累加进来；
- 最终加上常量表达式 `3 * i`。

总体而言，Java 层实现了一种依赖历史状态的前缀累加式混淆算法。

## Native 层逻辑

### 1. 魔改 Base64 解码

`NativeBridge.verify` 方法运行时会重新加载 `telemetry.bin`，并利用文件中提供的第 3 行（即 64 字节的自定义字符映射表），去解码第 4 行的目标字符串。

显然，程序采用的并非标准的 Base64 算法，而是一种替换了基础字典库的变种 Base64。

### 2. 动态密钥异或加密

在 native 环境下，程序并没有直接将 Java 层传来的数据进行硬核比对，而是对其施加了一层基于动态流密钥的异或运算。

该异或流的状态初始值设定为 `0x5d`，并在每一次循环中迭代更新。通过阅读反汇编指令，可知其本质上是一个 8 位的线性反馈移位寄存器（LFSR）：

```python
fb = ((state >> 2) ^ (state >> 3) ^ (state >> 5) ^ state) & 1
state = ((state >> 1) | (fb << 7)) & 0xff
k = (state + 7 * i) & 0xff
```

产生密钥后，执行如下操作：

```python
encoded[i] = transformed[i] ^ k
```

在完成上述所有的动态异或混淆后，程序才会把所得结果与前文变种 Base64 解码生成的目标字节序列进行最终对比。

## 逆向思路

综合上述分析，该题目的完整检验流程为：

```text
flag body
-> Java transform
-> native 动态异或
-> 与 telemetry.bin 目标字节比较
```

因此，我们需要采取逆向推导的策略来破局：

1. 首先，利用已知的替换表对第 4 行的密文执行变种 Base64 解码操作；
2. 其次，逆推 native 层的动态异或算法，从而获取到 Java 转换函数的原始输出流；
3. 接下来，针对 `CryptoEngine.transform` 的逻辑进行逐字节的反推运算；
4. 最终拼凑并还原出完整的 flag 字符串。

## 解题脚本

```python
import zipfile

apk = r"app-release.apk"
lines = zipfile.ZipFile(apk).read("assets/telemetry.bin").decode().splitlines()

shifts = bytes.fromhex(lines[0])
mask = bytes.fromhex(lines[1])
alphabet = lines[2]
target = lines[3]

alpha_map = {c: i for i, c in enumerate(alphabet)}

# 1. 自定义 Base64 解码
decoded = bytearray()
for i in range(0, len(target), 4):
    chunk = target[i:i + 4]
    vals = [0 if ch == "=" else alpha_map[ch] for ch in chunk]
    pad = chunk.count("=")
    n = (vals[0] << 18) | (vals[1] << 12) | (vals[2] << 6) | vals[3]
    decoded.append((n >> 16) & 0xff)
    if pad < 2:
        decoded.append((n >> 8) & 0xff)
    if pad < 1:
        decoded.append(n & 0xff)

# 2. 逆 native 动态异或
state = 0x5d
transformed = []
for i, b in enumerate(decoded):
    fb = ((state >> 2) ^ (state >> 3) ^ (state >> 5) ^ state) & 1
    state = ((state >> 1) | (fb << 7)) & 0xff
    k = (state + 7 * i) & 0xff
    transformed.append(b ^ k)

# 3. 逆 Java transform
prev = 0
body = []
for i, t in enumerate(transformed):
    sols = []
    for c in range(128):
        v = c
        if 32 <= v <= 126:
            v = ((v - 32 + shifts[i % len(shifts)]) % 95) + 32
        out = ((v ^ mask[i % len(mask)]) + prev + 3 * i) & 0xff
        if out == t:
            sols.append(c)
    assert len(sols) == 1
    body.append(sols[0])
    prev = t

body = bytes(body).decode("ascii")
print("ISCC{" + body + "}")
```

执行上述脚本后得到的结果为：

```text
ISCC{abyssal_vows_bind_the_last_protocol}
```
