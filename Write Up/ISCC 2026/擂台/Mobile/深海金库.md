
## 0. 题目背景与初步分析

题目提示：

> 深海金库。你在马里亚纳海沟的一艘废弃潜水器中发现了一个密封的金库。金库的显示屏上只有一行跳动的乱码和一张磨损的海螺照片。潜水器的操作指南上写着：“秘密深藏于耐心之中，而开启的旋律会随深度起伏。”

提供给我们的附件为一款安卓应用：`app-release-01.apk`。

针对这种 **Mobile 分支且附件为 APK** 的赛题，通常不能盲目尝试猜测 flag，建议遵循以下排查流程：

1. APK 文件实际上是 zip 压缩包，首先将其解压以观察内部目录结构；
2. 定位 Java/Kotlin 层的核心代码，即分析 `classes.dex` 文件；
3. 检查是否存在 native 层的 `.so` 动态库。若存在，往往意味着 Java 层仅作为外壳，核心校验逻辑被放在了 native 层；
4. 审查资源文件（如图片、多媒体或 assets 目录），思考题干里提到的“海螺照片”是否暗示了隐写术；
5. 寻找诸如用户输入框、flag 验证方法或 JNI 接口等关键点，并优先对这些验证流程进行逆向分析。

本题的整体验证逻辑链条如下：

```text
用户输入 flag
    ↓
Java 层检查 ISCC{...} 格式
    ↓
取出花括号内部内容
    ↓
Java 层做一次字符变换
    ↓
native 层做 Fibonacci 异或流
    ↓
native 层做自定义 Base64
    ↓
和目标字符串比较
```

因此，我们的解题思路即为将上述流程进行逆向推导。

## 6. FlagValidator 剖析：锁定 flag 格式及验证工作流

方法 `FlagValidator.isFormatValid()` 的功能仅限于格式把关：

```java
public static boolean isFormatValid(String input) {
    return input != null
        && input.startsWith("ISCC{")
        && input.endsWith("}");
}
```

通过该函数可以得知，合法的 flag 外观必须满足：

```text
ISCC{某段内容}
```

而 `FlagValidator.doFinalCheck()` 才是触及核心验证流程的关键入口：

```java
public static boolean doFinalCheck(MainActivity activity, String input) {
    String core = input.substring(5, input.length() - 1);
    byte[] transformed = CryptoEngine.performTransformation(core);
    return activity.nativeVerify(transformed);
}
```

这部分代码至关重要。它揭示了：

```text
完整输入：ISCC{xxxxxxxx}
取出的 core：xxxxxxxx
```

这就意味着，接下来所有的加密运算和数据比对，都仅仅围绕着花括号内的数据展开，排除了外壳 `ISCC{` 和 `}` 的干扰。

因此，我们接下来的逆向目标其实是：

```text
flag 中间的 core 部分
```

在解出核心内容后，再将其包裹还原：

```text
ISCC{core}
```

## 7. CryptoEngine 分析：解密 Java 层的数据变换

接下来查看 `CryptoEngine.performTransformation()` 函数。

经过逆向还原后的核心伪代码展示如下：

```java
public static byte[] performTransformation(String s) {
    byte[] arr = s.getBytes(StandardCharsets.US_ASCII);
    int prev = 0;

    for (int i = 0; i < arr.length; i++) {
        int x = arr[i] & 0xff;

        if (x >= 32 && x <= 126) {
            x = ((x - 25) % 95) + 32;
        }

        int delta;
        if (i % 2 == 0) {
            delta = 8;
        } else {
            delta = -8;
        }

        x = x + delta + (prev % 4);
        prev = x & 0xff;
        arr[i] = (byte) prev;
    }

    return arr;
}
```

针对上述逻辑，我们可以将其拆解为三个步骤来剖析。

### 7.1 步骤一：转换为 ASCII 字节序列

```java
byte[] arr = s.getBytes(StandardCharsets.US_ASCII);
```

此行代码表明，flag 的内部核心应该由常规的 ASCII 字符构成。

### 7.2 步骤二：针对可见字符的循环移位

```java
if (x >= 32 && x <= 126) {
    x = ((x - 25) % 95) + 32;
}
```

标准的 ASCII 可打印字符区间为：

```text
32 到 126
```

总共包含的字符数量为：

```text
126 - 32 + 1 = 95
```

由此可见，该步骤对这些可见字符执行了循环移位操作。通俗来讲就是：

```text
在 95 个可打印字符组成的环上移动
```

这并不是常规的哈希（如 MD5）或加密算法（如 AES 或标准 Base64），仅仅是基于字符集内部的一种轮换机制。

### 7.3 步骤三：结合索引与前置状态的混淆扰动

```java
int delta = (i % 2 == 0) ? 8 : -8;
x = x + delta + (prev % 4);
prev = x & 0xff;
```

此处存在两个需要留意的细节：

1. 当索引为偶数时，偏移量增加 `8`；若为奇数，则减去 `8`；
2. 当前字节的运算结果与上一次迭代产生的值 `prev` 息息相关。

由于当前字符的值受到历史输出的影响，这使得字符无法被独立解析。不过，鉴于 `prev` 代表前一次的输出结果，且在逆推过程中我们完全可以获取到目标密文的前序字节，因此依然能够按照从左至右的顺序逐一逆向破解。

至此，Java 层的整个数据变换流程已被完全摸清。

## 8. 深入 native so 的必要性

注意到 Java 层的代码最终执行了如下调用：

```java
activity.nativeVerify(transformed)
```

这说明，Java 层面仅仅是充当了一个数据搬运工，将转换后的字节流透传给 native 层，其自身并未包含最终结果的校验逻辑。

基于此，接下来的顺理成章之举便是对 `libsecure_verify.so` 展开分析。

该 APK 包内提供了适配多种处理器架构的 so 文件：

```text
lib/arm64-v8a/libsecure_verify.so
lib/armeabi-v7a/libsecure_verify.so
lib/x86/libsecure_verify.so
lib/x86_64/libsecure_verify.so
```

这些库文件内部的代码逻辑本质上是一致的，差异仅在于目标指令集。考虑到静态分析的便利性，建议挑选 `x86_64` 架构的库文件进行分析，毕竟 x86_64 的汇编指令对于初学者而言可读性更强。

尽管这些 C++ 函数的名称经过了名称修饰（name mangling）处理，我们依然可以推测出它们的具体用途：

| 符号名称 | 功能推测 |

回想一下赛题背景中的那段话：

> “秘密深藏于耐心之中，而开启的旋律会随深度起伏。”

结合在此处发现的 `process_fib_stream` 函数，这句描述中的“旋律”与“起伏”便有了着落：它隐喻了某种呈现出周期性或序列化波动的操作流，而这里使用的正是斐波那契数列。

## 10. native 层的核心验证逻辑 SV::v

采用 IDA 或 Ghidra 等可视化工具分析 so 文件会更加直观。如果习惯使用命令行，可以执行：

```bash
objdump -d -Mintel apkwork/lib/x86_64/libsecure_verify.so > so.asm
```

结合之前 `readelf` 提供的线索，`SV::v` 函数大致位于 `0x215f0` 这个地址。将其核心逻辑进行反编译与抽象，可以表示为以下伪代码：

```cpp
bool SV::v(unsigned char const* input, int len) {
    if (input == nullptr || len <= 0) {
        return false;
    }

    vector<unsigned char> buf(input, input + len);

    process_fib_stream(buf.data(), len);

    string encoded = encode_b64(buf);

    string target = "cUdltutGeWWI";
    target.append("Um+7OE4ce3wi");

    return encoded == target;
}
```

这段代码揭示了两个重要信息。

其一，native 层并没有将 Java 层传入的数据直接拿来比对，而是经历了如下流水线：

```text
Java 变换后的字节
    ↓
Fibonacci 异或流
    ↓
自定义 Base64 编码
    ↓
和目标字符串比较
```

其二，最终用于比对的目标密文并没有作为一个完整的字符串硬编码在 `.rodata` 节中，而是被分割成了两块：

```text
cUdltutGeWWI
Um+7OE4ce3wi
```

合并后的完整形态为：

```text
cUdltutGeWWIUm+7OE4ce3wi
```

将字符串切片处理是 CTF 赛题中常用的轻量级混淆手段。单纯依赖 `strings` 命令往往无法截获完整的密文，但在反汇编代码中，这种拼接逻辑便会原形毕露。

## 11. 解析 process_fib_stream：斐波那契异或混淆

`process_fib_stream` 函数构成了本地 C++ 代码中极其关键的一环。

通过阅读汇编指令，能够观察到其维护了两个字节大小的状态变量：

```asm
mov cl, 0x1
mov dl, 0x1
...
xor BYTE PTR [rdi+rsi], cl
xor BYTE PTR [rdi+rsi+1], dl
add cl, dl
xor BYTE PTR [rdi+rsi+2], cl
add dl, cl
xor BYTE PTR [rdi+rsi+3], dl
```

将其转换为更加直白的高级语言伪代码如下：

```cpp
void process_fib_stream(unsigned char* data, int len) {
    unsigned char a = 1;
    unsigned char b = 1;

    for (int i = 0; i < len; i++) {
        data[i] ^= a;
        unsigned char c = a + b;
        a = b;
        b = c;
    }
}
```

该算法产生的异或密钥流表现为：

```text
01 01 02 03 05 08 0d 15 22 37 59 90 e9 79 62 db 3d 18 ...
```

必须留意的是，由于运算是在单字节范围内进行的，一旦发生溢出，超出 `0xff` 的部分将会被自动舍弃，这等同于执行了对 256 取模（`% 256`）的操作。

鉴于异或（XOR）运算具备对称性，此类混淆的逆向还原十分轻松：

```text
A xor K = B
B xor K = A
```

也就是说，解密过程与加密过程完全共用同一套逻辑。

## 12. encode_b64：魔改版 Base64 编码

`encode_b64` 乍看之下是常规的 Base64，但仔细甄别后会发现，其底层的编码字典并非官方标准。

我们知道，常规的 Base64 字典长这样：

```text
ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/
```

然而，通过分析该原生库得知，其采用的自定义字典为：

```text
zKJUExRaVtM3Ydv5TQIsWD1frnHC78Lckl6euPh9AGoj0SgN4Zp+OwXi2F/ybmBq
```

要挖掘出这串魔改字典，通常有两条路径。

第一种途径：利用命令全局检索 so 文件内部的长文本串：

```bash
strings -a apkwork/lib/x86_64/libsecure_verify.so | grep "zKJ"
```

第二种途径：直接借助 IDA 或 Ghidra 等工具，追踪 `encode_b64` 函数内部对数据段的引用。

虽然字典被替换，但 `encode_b64` 的内部位运算机制仍与原生 Base64 保持同步：

```text
3 个字节分成 4 组 6 bit
每组 6 bit 当作下标
去自定义 alphabet 里取字符
最后不足 3 字节时补 =
```

基于这个原理，我们在编写逆向脚本时无需进行穷举爆破，只需将标准解码器中的字典替换为上述变种字典即可完成解码。

## 13. 梳理全局正向执行流

假定用户输入的原始文本为：

```text
ISCC{core}
```

整个系统验证输入数据的正向路线可总结为：

```text
core
  ↓ Java: CryptoEngine.performTransformation
java_transformed_bytes
  ↓ native: process_fib_stream XOR Fibonacci
fib_processed_bytes
  ↓ native: custom Base64 encode
encoded_string
  ↓ compare
cUdltutGeWWIUm+7OE4ce3wi
```

为了获取正确答案，我们需要执行逆向回溯，步骤则刚好反过来：

```text
cUdltutGeWWIUm+7OE4ce3wi
  ↓ 自定义 Base64 解码
fib_processed_bytes
  ↓ Fibonacci XOR 还原
java_transformed_bytes
  ↓ 逆 Java 层变换
core
  ↓ 拼接格式
ISCC{core}
```

在解决这类逆向工程问题时，必须牢记一条核心准则：

> 绝不能靠盲目猜测来试探答案，正确的方法是精准描绘出所有的正向运算节点，再以最终的目标密文为起点，步步为营地反向推导。

---

## 14. 逆向步骤一：执行变种 Base64 解码

我们要破解的密文目标是：

```python
target = "cUdltutGeWWIUm+7OE4ce3wi"
```

前面提取到的非标准字典为：

```python
alphabet = "zKJUExRaVtM3Ydv5TQIsWD1frnHC78Lckl6euPh9AGoj0SgN4Zp+OwXi2F/ybmBq"
```

对应的解码程序如下：

```python
def custom_b64_decode(s, alphabet):
    out = bytearray()

    for i in range(0, len(s), 4):
        block = s[i:i + 4]
        nums = []

        for ch in block:
            if ch == "=":
                nums.append(0)
            else:
                nums.append(alphabet.index(ch))

        triple = (nums[0] << 18) | (nums[1] << 12) | (nums[2] << 6) | nums[3]

        out.append((triple >> 16) & 0xff)
        if block[2] != "=":
            out.append((triple >> 8) & 0xff)
        if block[3] != "=":
            out.append(triple & 0xff)

    return bytes(out)
```

运行后提取到的中间阶段数据为：

```text
7c 33 61 26 42 69 8d 45 12 0f dc dc d0 4c 1f 8c bd 77
```

需要指出，这并非我们苦苦寻觅的 flag 核心，它仅仅是斐波那契异或处理之后留下的产物。

## 15. 逆向步骤二：撤销 Fibonacci 异或扰动

正如前文所述，加密时的异或操作为：

```text
data[i] ^= fib[i]
```

解密时依然遵循同样的法则：

```text
data[i] ^= fib[i]
```

相关的还原代码如下：

```python
def fib_xor(data):
    data = bytearray(data)
    a = 1
    b = 1

    for i in range(len(data)):
        data[i] ^= a & 0xff
        a, b = b, (a + b) & 0xff

    return bytes(data)
```

将上个步骤获得的产物馈入该函数，便能反推出 Java 层处理完毕后交接给 JNI 的数据流：

```text
7d 32 63 25 47 61 80 50 30 38 85 4c 39 35 7d 57 80 6f
```

这串数据依然不能直接使用，它是 `CryptoEngine.performTransformation(core)` 函数经过运算后吐出的内容。

## 16. 逆向步骤三：剥离 Java 层的数据混淆

回顾前文解析出的 Java 层正向操作逻辑：

```python
x = 原字符 ASCII

如果 x 是可打印字符：
    x = ((x - 25) % 95) + 32

如果 i 是偶数：
    delta = 8
否则：
    delta = -8

x = x + delta + (prev % 4)
prev = x & 0xff
输出 prev
```

此时，该操作输出的每一位数据 `y` 已经暴露在我们面前：

```text
7d 32 63 25 47 61 80 50 30 38 85 4c 39 35 7d 57 80 6f
```

考虑到 `prev` 变量记录的是上一个生成的结果，而这些结果序列已被我们完全掌控，因此直接从左向右顺序解密是完全可行的。

为了降低出错率，与其费力去推导复杂的数学逆运算公式，不如采用更具鲁棒性的暴力枚举策略，穷举所有潜在的可见字符：

```python
for ch in range(32, 127):
    # 把 ch 走一遍 Java 正向逻辑
    # 如果结果等于当前 y，就说明 ch 是这一位原字符
```

利用这种穷举法，能够完美规避掉 Java 中诸如取模运算的边界、字节溢出截断以及负数处理等诸多暗礁。

## 17. 综合解码脚本与结果展示

以下整合了各步骤逻辑的全局破解脚本。为了契合题解说明规范，脚本的输出环节对最终的 flag 进行了脱敏掩码处理。

```python
alphabet = "zKJUExRaVtM3Ydv5TQIsWD1frnHC78Lckl6euPh9AGoj0SgN4Zp+OwXi2F/ybmBq"
target = "cUdltutGeWWIUm+7OE4ce3wi"


def custom_b64_decode(s, alphabet):
    out = bytearray()

    for i in range(0, len(s), 4):
        block = s[i:i + 4]
        nums = []

        for ch in block:
            if ch == "=":
                nums.append(0)
            else:
                nums.append(alphabet.index(ch))

        triple = (nums[0] << 18) | (nums[1] << 12) | (nums[2] << 6) | nums[3]

        out.append((triple >> 16) & 0xff)
        if block[2] != "=":
            out.append((triple >> 8) & 0xff)
        if block[3] != "=":
            out.append(triple & 0xff)

    return bytes(out)


def fib_xor(data):
    data = bytearray(data)
    a = 1
    b = 1

    for i in range(len(data)):
        data[i] ^= a & 0xff
        a, b = b, (a + b) & 0xff

    return bytes(data)


def java_forward_one(ch, i, prev):
    x = ch

    if 32 <= x <= 126:
        x = ((x - 25) % 95) + 32

    if i % 2 == 0:
        delta = 8
    else:
        delta = -8

    x = x + delta + (prev % 4)
    return x & 0xff


def reverse_java(transformed):
    result = []
    prev = 0

    for i, y in enumerate(transformed):
        candidates = []

        for ch in range(32, 127):
            if java_forward_one(ch, i, prev) == y:
                candidates.append(ch)

        if len(candidates) != 1:
            raise RuntimeError(f"第 {i} 位候选数量异常: {candidates}")

        result.append(chr(candidates[0]))
        prev = y

    return "".join(result)

def mask_flag(flag):
    # 只展示前缀和一小段，隐藏核心答案
    left = flag[:11]
    right = flag[-2:]
    return left + "*" * (len(flag) - len(left) - len(right)) + right


stage1 = custom_b64_decode(target, alphabet)
print("custom b64 decode:", stage1.hex(" "))

stage2 = fib_xor(stage1)
print("after fib xor:", stage2.hex(" "))

core = reverse_java(stage2)
flag = "ISCC{" + core + "}"

print("masked flag:", flag)
```

执行上述代码，控制台将输出如下信息：

```text
custom b64 decode: 7c 33 61 26 42 69 8d 45 12 0f dc dc d0 4c 1f 8c bd 77
after fib xor:     7d 32 63 25 47 61 80 50 30 38 85 4c 39 35 7d 57 80 6f
masked flag:       ISCC{n2R#7_pQ!9vL*5mWnp}
```

## 19. 针对移动端逆向题的实战经验总结

通过本题，我们可以沉淀出一套解决 Android 逆向题目的标准化 SOP。

### 19.1 第一步：摸清内部目录

```bash
unzip app.apk -d out
find out -maxdepth 3 -type f
```

目光应主要聚焦于：

```text
classes.dex
classes2.dex
lib/*.so
assets/
res/
```

### 19.2 第二步：依靠 strings 敏锐嗅探

```bash
strings -a classes.dex | grep -E "flag|Flag|check|verify|native|ISCC|CTF"
strings -a lib/xxx.so | grep -E "flag|verify|base|xor|key|target"
```

切莫轻视 `strings` 的威力。它屡建奇功，能快速为你锁定敏感类、关键方法、成功后的祝贺语乃至关联的底层库。

### 19.3 第三步：筛选 Java 层的核心方法

需重点关注以下三类目标：

1. 捕获用户 UI 交互（如文本框输入或按钮点击）的监听器；
2. 负责合法性筛选的方法，例如 `startsWith("ISCC{")`；
3. 负责核心校验或数据演化的模块，诸如带有 `check`、`verify`、`validate` 或 `CryptoEngine` 字眼的标识。

### 19.4 第四步：循着 native 接口深入底层

倘若在 Java 源码中发觉类似踪迹：

```java
System.loadLibrary(...)
public native boolean nativeVerify(...)
```

这就发出强烈信号：真正的判定中枢已脱离 DEX 文件，深埋于 so 动态库之中。

### 19.5 第五步：探测 native 层的抛头露面者

```bash
readelf -Ws libxxx.so | grep -E "verify|check|encode|decode|xor|JNI"
```

只要符号表没有被 strip 抹除得一干二净，光看函数名就能获取到海量的情报。

### 19.6 第六步：绘制全局状态流转图

以本案为例，其正向的数据加工流水线为：

```text
core
→ Java printable rotate
→ Java odd/even delta + prev state
→ Fibonacci XOR
→ custom Base64
→ compare target
```

一旦厘清了这张流程图，反向编写解密程序便是水到渠成。

### 19.7 第七步：借助程序化思维替代人工推演

依赖人工计算极易翻车，特别是在面对以下场景时：

- 字节容量溢出；
- 基于 256 的取模运算；
- Java 中特有的有符号字节（signed byte）坑；
- 遭遇非标准的 Base64 字典；
- 运算中带有强烈的上下文依赖（如前后文状态关联）。

因此，更为妥当的策略是：

```text
能写正向函数，就写正向函数；
逆不出来时，就枚举候选字符，看谁走正向能对上。
```

我们在应对本题 Java 层的逆向解析时，便充分践行了这一指导思想。
