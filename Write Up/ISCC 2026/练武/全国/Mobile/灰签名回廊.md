
### 1. 题目概览
本程序的 UI 界面非常简洁，仅包含文本输入区域以及一个用于触发验证的按钮。然而，内部真正的校验逻辑却被巧妙地分布在了 Java 层逻辑、数个隐藏的 Android 组件以及底层的 native 动态库中。其中发挥骨干作用的 Java 类系 `com.example.gnd.security.SecureValidator`，涉及到的隐藏组件涵盖了 `SecretContentProvider`、`SecretReceiver` 还有 `SecretActivity`。而底层的 native 层验证业务则位于 `libnative-verify.so` 动态链接库内。

`SecureValidator.c(String)` 方法首先会检查输入的文本形式，接着将包裹在大括号内部的 24 字节数据剥离，并按固定规律拆分成四个部分：长度依次为 7、5、5、7。这四个子字符段会被分别传入 `nativeVerifyPart1/2/3/4` 函数进一步实施拦截判断，最终还必须保障整条 flag 计算出的 SHA-256 哈希值与给定的哈希常量吻合。

| 指标项目 | 数据内容 |
| :--- | :--- |
| 输入格式 | `ISCC{<24 chars>}` |
| body 分段长度 | part1=7, part2=5, part3=5, part4=7 |
| 预置常量种子 | `0x5f3759df` |
| native magic 校验 | 12；body 长度 = 12 * 2 = 24 |

### 2. Java 层格式与整体哈希校验
在 Java 层的工作流中，程序利用了一个整型数组 `[88,66,82,82,106]` 逐一与 `0x11` 实施异或运算，从而推导得出 `ISCC{` 这个标志前缀。紧接着限制整个输入数据流必须以 `}` 符号收尾，此时截取出 `flag[5:-1]` 作为长度定为 24 字节的核心校验主体（body）。此外，通过调用 `b(String)` 方法，程序还会对完整的 flag 数据串执行一次 SHA-256 哈希计算，期待达成匹配的一致性结果如下：
`0dad698c7946f007ee28673684b78767227986fc6aa13ddbe3ac0becb1a588e7`

### 3. 四部分 Key 的来源路径

| 指标项目 | 数据内容 |
| :--- | :--- |
| part1 key | APK v2 签名证书 SHA-256；对应的证书 hash = `133415c3f8aad739be8d0df905481208aa7448da7254a474028eeeddfc2849af` |
| part2 key | 从 `SecretContentProvider` 取得: `[103,47,70,82]` ^ `0x17` => `p8QE` |
| part3 key | 从 `SecretReceiver` 取得: `(index ^ value ^ 0x5A)`，其中 `value=[0x1f,0x2e,0x3f,0x01]` => `EugX` |
| part4 key | 由 `SecretActivity` 提取: 字符串 `"ISCC2024S"` 的第 7、8 字节提取 => `4S` |

### 4. Part 1 逻辑逆推反解
在仔细追溯 `nativeVerifyPart1` 的正常分支时，可发现它对我们提交数据的每一个单一字节都实施了 `((ch + 1) ^ 0x07) ^ sigHash[i % 4]` 形式的处理逻辑。考虑到回到 Java 层其设定的目标匹配公式为 `const ^ sigHash[i % 4]`，这就使得签名的哈希数值能够在比较环节的两边互相消除掉。

```python
const = [0x6e, 0x7f, 0x7f, 0x4b, 0x65, 0x7c, 0x53]
part1[i] = ((const[i] ^ 0x07) - 1) & 0xff
part1 = "hwwKazS"
```

### 5. Part 2 与 Part 3 逻辑逆推反解
针对 Part 2 部分可知其派发密钥系 `p8QE`，深入观察 native 库对于头一个字节以及配合 SIMD shuffle 技术的判断规则，我们可以非常直接地通过逆运算得出明文本质为 `_GPVu`。

```python
part2[0] = ord('p') ^ 0x2f = '_'
[p4, p1, p2, p3] ^ b"p8QE" = [0x05, 0x7f, 0x01, 0x13]
part2 = "_GPVu"
```

对应于 Part 3 所关联的特征密钥为 `EugX`。在其 native 处理流程里，此步骤面向每个特定的比特位均增设了基于 `(ch ^ 0x19) + key_byte` 的一道线性判定式。

```python
key = b"EugX"
targets = [0x8b, 0xbf, 0xc4, 0xa5, 0x92]
part3 = bytes(((target - key_byte) & 0xff) ^ 0x19 for ...)
part3 = "_SDTT"
```

### 6. 链式认证 Token 生成及 Part 4 分段处理
在获得前面三块的答案后，代码调用 `nativeComputeChainToken` 方法进一步改变全局标量 `c` 的状态。此处的函数内核实质上相当于构建 了一套附有位循环转移 (rotate) 能力的类 FNV 散列搅动结构，该结构运算所得的凭证直接被作为原料去生成后续赋给 Part 4 用于工作的 RC4 因子。

```c
FNV = 0x100000001b3
h = ((part * FNV) & ((1<<64)-1)) ^ seed
for byte in data:
    h = rol64(((h ^ byte) * FNV) & mask, 13)
h ^= h >> 33
h = (h * 0xff51afd7ed558ccd) & mask
h ^= h >> 33
```

沿着上述轨迹推演到底，我们可以得出 `h3 = 0xd1b5f240969f3de4`。进而推向 part4，拿到了组装它的主干料（key material）：
`b758fce370385145457567580000`

针对所用的 RC4 算法，其初始 4 个字节长度的密码流 (keystream) 是：
`3b8aa23a`

对于 `nativeVerifyPart4` 方法，审查表明它其实只检测了当前第四阶段块落入的前四个字节，所需达标公式在下边展示；至于剩下的 3 字节缺失部分，则需通过联合计算最终总的 SHA-256 以执行高频强力试探来解密。

```python
a = p4[0] ^ ks[0]
b = p4[1] ^ ks[1]
c = p4[2] ^ ks[2]
d = p4[3] ^ ks[3]
((((c ^ (b << 3) ^ (a << 6)) << 12) ^ (d << 9)) & 0x03ff0000) == 0x01ed0000
```

### 7. 尾部 7 字节推导获取
在前文确定了开头的三个片段之后，flag 字串的先头内容已知即为 `ISCC{hwwKazS_GPVu_SDTT`。剩下面临的最后那截由 7 个字节构成的未知子串，允许借助正则集合 `[0-9a-zA-Z_]` 实施枚举匹配。在这个阶段，我们先通过前述提炼的 Part 4 原生设定进行四字节拦截过滤，然后依赖 OpenSSL 提供的方法 `SHA256_Transform` 确保完成品与大前提严格吻合。

```python
known_prefix = b"ISCC{hwwKazS_GPVu_SDTT"
suffix = b"_hoigXW"
flag = known_prefix + suffix + b"}"
sha256(flag) = 0dad698c7946f007ee28673684b78767227986fc6aa13ddbe3ac0becb1a588e7
```
