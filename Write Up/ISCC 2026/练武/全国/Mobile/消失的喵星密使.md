首当其冲，我们需要对 APK 执行常规的反编译与解压拆包操作：

随后能够锁定以下关键性文件及程序入口：

  - 包名：`com.example.mobile03`
  - 启动页：`com.example.mobile03.MainActivity`
- Java 层面关键逻辑类：
  - `temp/meowl_jadx/com/example/mobile03/MainActivity.java`

- Native 核心动态库：

可以看到 Java 层的代码量很少，核心业务流程集中在 `MainActivity` 之中：

1. 初次加载页面时，应用会弹出一系列剧情提示；
2. 界面上存在一个 6x6 的 Emoji 图标矩阵，当用户点击时，`_B._q()` 方法会将二维触控坐标编码为 2 个字母，并依次追加拼接到一起；
3. 接着 `_C._c()` 会对产出的步态字符串进行 MD5 哈希校验，计算值需完全吻合固定常量；
4. 约束提交给输入框的字符内容需要严格按照 `ISCC{...}` 这一规范格式包裹；
5. 最终经由 `_A.check()` 接口，将下列三个关键参数：
   - flag 涉及的内核明文
   - 之前生成的步态字符串
   - 资源文件 `hint.png`
   共同传入底层 JNI 方法 `_v(...)` 以供决断。

由此推断，我们真正需要攻克的难点模块其实包含两个：

- **根据约束推导出步态字符串 `q`**
- **逆向分析 native 层的图片隐藏数据以及高度定制化的加密算法**

## 详细解题过程

### Step 1: 摸清 Java 表层门槛规则

我们首先查看 `MainActivity.java` 的主验证逻辑：

```java
String str_q = this._b._q();
if (!_C._c(str_q)) {
    Toast.makeText(this, "步伐的韵律不对，密使已消失。", 0).show();
    this._b._r(gridLayout);
    return;
}
String str_g = this._b._g((EditText) findViewById(R.id.et_flag));
if (str_g == null) {
    Toast.makeText(this, "格式错误！", 0).show();
} else if (this._a.check(this, str_g, str_q)) {
    Toast.makeText(this, "喵！解密成功！", 1).show();
}
```

从上述代码中我们能够精准得出三点结论：

1. **所有的验证前提在于步态字符串必须先穿透 MD5 校验的这层护盾**
2. **应用层的校验对 flag 的外套限制固定为了 `ISCC{...}`**
3. **决定解密成败的深水区全在 `_A.check()` 其挂载的 native 方法 `_v(...)` 内部**

进一步探究，可以发现 `_B._g()` 包含了一段极简的流密码按位异或操作：

```java
byte[] bArr = {11, 17, 1, 1, 57};
for (int i = 0; i < 5; i++) {
    sb.append((char) (bArr[i] ^ 66));
}
```

异或还原出的前缀便是 `ISCC{`，这就证明了外在包裹格式并非本次挑战的重心。

与此同时，`_C._c()` 组件对传入字符串 `q` 执行的 MD5 判断彻底被硬编码卡死：

```java
return sb.toString().equals("72184e58e765a1a579b201c3e15bc0ce");
```

因此，我们解密路径的第一阶段首要任务转变为：

> **通过各方线索，复原出那串总计 12 位的正确步态序列 `q`**

### Step 2: 结合内部规则文书与页面文案逆推步骤

存放于 `assets/sys/OBSERVER_PROTOCOL` 的协议文件实质上并非普通纯文本存放，而是被转换成了十六进制 ASCII 。将它解码反向转换回来便能阅读到关键信息：

```text
[GLOBAL_SEED]: 0xDEADC0DE
[RUNTIME_STR]: @LOG_TAG_DESCRIPTOR
[CONSTRAINT]: S = { x | 1 ≤ x ≤ 6, x ∈ ℤ }

[[ PHASE_ALPHA: THE_ORIGIN_BIAS ]]
...
P1.r = r_A; P1.c = c_A / λ

[[ PHASE_BETA: THE_FRACTAL_SHIFT ]]
...
r_B = τ + (φ >> 31) + 1
c_B = ((r_B << τ) ^ (T & 0x00)) & 0x07

[[ PHASE_GAMMA: THE_ASYMPTOTIC_GHOST ]]
...
r_C = lim(x→∞) [ (6x² + 5x) / (x² + 1) ]
c_C = ⌈ cos(π) ⌉ + ghost + 2

[[ GLOBAL_PARITY_CHECK ]]
∫ Row_Total dt  ≡ 21  (Mod 100)
∫ Col_Total dt  ≡ 20  (Mod 100)
隐藏条件：当且仅当 P2.c > P1.c 且 P6.c < P2.c 时，序列激活
```

这些公式乍看之下颇有威慑力，但经过审视实则极其公式化：

#### 2.1 依赖简单数理运算获悉前两个坐标

审视 `PHASE_ALPHA` 的规则定义：

- 给定的数学函数等式为 `Ψ(r, c) = (r - c)^2 + (r + c - 2)^2`
- 计算其驻点极值可迅速求出 `r = 1, c = 1`

因此起始的第一个触控坐标确定为 **(1,1)**。

回顾页面实现中映射这块网格图的行列查表是由 `_B._k` 建立的：

```java
private final String[] _k = {"D", "F", "X", "G", "A", "V"};
```

替换后可以得知 `(1,1)` 解析后对应的密标为 `DD`。恰好在 UI 当中左上角摆放的也是 `🐾` 元素，与日志描述中提到的“印记的起点”无缝照应。

转而看 `PHASE_BETA` 的算子要求：

- 获取 `_P` 类内部定义的 log tag，能够发现其存在常量 `"0x1918"`
- 对此赋值，演算 `τ = (0x1918 >> 12) & 0x0F = 1`
- 进而得出行的游标 `r_B = 1 + 0 + 1 = 2`
- 以及列的游标 `c_B = ((2 << 1) ^ 0) & 0x07 = 4`

至此第二个踏入的点即为 **(2,4)**，其映射编码将是 `FG`。而网格中这一格放置的恰是 `🦋`，对应的也是剧情文案中“舞动的斑斓”一词。

#### 2.2 借助明面剧情文本补平剩下的按键顺序

UI 前台给出的猫咪故事背景如下：

```text
密使巡逻日志：雪球在印记的起点打转，它转而扑向那抹舞动的斑斓，在清甜的气息中小憩片刻；它凝视苍穹中央的光源，却为未竟之事而黯然，终没入无光的轮廓里。
```

将这番颇具童话感的文本结合界面自带的 Emoji 清单一看便知道，它等同于是在按序列指挥你去触发某些特定图标。行文来到第三步描绘时，往往大家很容易想岔到 `🥛` 或者 `🛌` 身上，然而如果死板照做，将提取出来的坐标带入底层协议强制附带的校验条件 `Row_Total = 21` 和 `Col_Total = 20` 时就会验算错误；反观之下，使用具有猫薄荷或香草意味的 `🌿` 图标才能够双向兼顾逻辑含义和奇偶总和。

在此指引下，正确的交互长跑链路被唯一固化：

- 印记的起点 → 敲击 `🐾` 处 → 原位坐标 `(1,1)` → 转义为 `DD`
- 舞动的斑斓 → 敲击 `🦋` 处 → 原位坐标 `(2,4)` → 转义为 `FG`
- 清甜的气息 → 敲击 `🌿` 处（最完美的意象投射） → 原位坐标 `(3,6)` → 转义为 `XV`
- 苍穹中央的光源 → 敲击 `☀️` 处 → 原位坐标 `(4,2)` → 转义为 `GF`
- 黯然 → 敲击 `😿` 处 → 原位坐标 `(5,6)` → 转义为 `AV`
- 没入无光的轮廓 → 敲击 `🐈‍⬛` 处 → 原位坐标 `(6,1)` → 转义为 `VD`

将提取出来的这个坐标序列罗列，可以看出它内嵌的校验数学美感：

- 全部的行数加总：`1 + 2 + 3 + 4 + 5 + 6 = 21`
- 全部的列数加总：`1 + 4 + 6 + 2 + 6 + 1 = 20`

此项加和表现完全适配了协议文书里的整合校验(parity check)法则。

通过这一段破译流程，整条字符串便能稳稳地拼凑而就：

```text
q = DDFGXVGFAVVD
```

我们可以利用一段便捷脚本复测下此字符串散列后的值是否吻合条件：

```bash
python3 - <<'PY'

q='DDFGXVGFAVVD'
print(hashlib.md5(q.encode()).hexdigest())
PY
```

打印出的结果展现为：

```text

```

完全相同，此刻宣告我们在 Java 环境下的防御封锁线已被全部剪除。

### Step 3: Native 层的深度行为揭秘

将 `libmobile03.so` 丢进逆向工具查看动态符号导出，最核心的动作全交织在这几个片段：

```text
_f_val(_JNIEnv*, _jobject*, _jstring*, _jstring*, _jobject*)
_f_ext(_JNIEnv*, _jobject*, unsigned char*)
_f1(char const*, unsigned char*)
_f_exp(unsigned char const*)
_f_enc(unsigned char*)
_f_d64(char const*, unsigned char*)
_f_monitor(void*)
```

#### 3.1 意图驱离调试者的 `_f_monitor`

观察它 `JNI_OnLoad` 的初始过程，可发现其显式抛出 `pthread_create` 孵化了一根指向 `_f_monitor` 的专属心跳检测线程，该线程犹如眼线一般反复从 `/proc/self/status` 之中探测 `TracerPid:` 所处状态，一旦识别该条目逃匿了数值 0，程序马上会触发 `_exit` 中止自身进程。

此举警示我们：

- 如果盲目附加挂钩开展动态追踪只会深陷泥潭；
- 恰恰反过来，**单凭纯静态的反编译和研判完全能够驾驭全场**，进而帮助我们避开和反调试功能正面硬抗的高昂代价。

#### 3.2 借助 `_f_ext` 解析出图片像素深处的 256 字节魔改 S-box

这里剖解 `_f_ext` 运转原理十分关键：

1. 借助标准接口 `AndroidBitmap_getInfo` 与 `lockPixels` 来提取传入那幅图的所有像素矩阵数据；
2. 按照行逐序去步进扫描每个象素；
3. 将着眼点投送到每个有效像素内的第 3 位字节（偏移位于 `+2`）剥出它的最低位 LSB 标记位；
4. 源源不断地提炼靠前的 `0x830` 组 bit 信息量，进而揉成总额达 `0x106 = 262` 的比特字节流；
5. 打包成的这一串比特必须符合 `MEOW` 这 4 比特魔数定界符做头校验；
6. 最后紧贴在包头之后的那 256 个字节被拷贝并注册为后续所用的一套查表矩阵（S-box）。

很显然，此处的 `hint.png` 被塞入的不单单是肉眼可见的色彩信息或明文，而是深度篡改利用了 LSB 隐写术，将下列结构固化进去：

```text
MEOW || SBOX[256] || 2 extra bytes
```

而我们私下编写脱机提取脚本抽丝剥茧看到的数据起始头也印证了这一点：

```text
4d454f57b5d3147d...
```

若用 ASCII 反解开就是：

```text
MEOW b5 d3 14 7d ...
```

#### 3.3 透视 `_f1` 函数的奇妙变换：组装 32 字节衍生密匙

在生成用来挂靠算法的解密秘钥上，`_f1` 函数并不草率地对待输入的 `q`，而是要对其施行了一套行列坐标重排戏法。

将原态字符串 `q = DDFGXVGFAVVD` 作为原料抛入，算法将会按照以下的索引规律抽调：

- 取出它第 `1,5,9` 顺位的标记
- 取出它第 `0,4,8` 顺位的标记
- 取出它第 `2,6,10` 顺位的标记
- 取出它第 `3,7,11` 顺位的标记

这种交错整合使得字符串重构成了另一个序列：

```text
DVVDXAFGVGFD
```

在此基础上，程序按照成双成对的分拆法则组合字母，继而投入一张规格为 6x6 的查询表展开寻址比对分析：

```text
mh7q1a
u6sl5p
cv20t9
zedwxf
njrgi8
bk4y3o
```

令人感到饶有趣味的是，上述阵列布局精准切中了存在于 `_f1` 只读段区中的一串长达 36 位固定预埋字典：

```text
mh7q1au6sl5pcv20t9zedwxfnjrgi8bk4y3o
```

这道工序执行完毕后，系统在 Native 环境内将该字母赋予的 ASCII 整数当作引索游标，投递进刚才剥除了图片伪装的特定 S-box 内进行一次替换拿取数据，此外还额外叠加了一层步幅跨度不变的异或层级：

```text
counter = 0x00, 0x13, 0x26, 0x39, ...
```

综合上述极其精细的连贯步骤，终于产出了主轴上的那个 32 字节 Key：

```text
44a17d9e2cb83637c30cde36a04551ba50a412db2728c2528c69b5a674c07eff
```

### Step 4: 剖离 `_f_exp + _f_enc` 本身，实为套壳换骨式的 AES-256 分组算法

这片算法腹地是题目出神入化的地方。

#### 4.1 密钥规划扩展引擎 `_f_exp`

深入审阅 `_f_exp` 代码架构发现，它的底座完全建立在经典 AES-256 Key Expansion 模型之上：

- 种下的最起始基础秘钥要求维持 32 字节宽幅；
- 它的轮更迭系数保持了最权威标准范式的 `Rcon = 01 02 04 08 10 20 40 80 1B 36` 散布；
- 而最大的分水岭被设计在 `SubWord` 步骤内，程序在字代换运算上摒弃了原生 AES 提供的白盒置换表，而是导入并使用了那本夹杂在 `hint.png` 里的那套自制版 S-box 查阅图谱。

#### 4.2 数据块混淆模块 `_f_enc`

对准传入加密大关的每个均等的 16 字节容量片段，`_f_enc` 会依次对它们下发如下处方集：

1. 先注入一轮 AddRoundKey(初始阶段)
2. 铺开实施重复 13 轮的核心工序网：
   - 依赖魔改版字表实现 SubBytes (字节代换)
   - ShiftRows (行移位处理)
   - MixColumns (按列混淆演算)
   - AddRoundKey (并入轮次新密钥)
3. 打底扫尾的局终一环，遵循规制不再触发 MixColumns 动作

抽丝剥茧到了这一步，底层机理昭然若现：

> **本质完全就是一个套嵌了深度自研 S-box 替换组件的 AES-256 ECB 加密模块！**

#### 4.3 `_f_val` 所守护的裁判比较目标

总控枢纽 `_f_val` 循规蹈矩地完成了以下一整串操作审批：

1. 第一条流指令从 `_f_ext` 中切走隐图里的特质 S-box
2. 借力联动 `_f1` 和 `_f_exp` 以便吸收并从字元串 `q` 这里衍生铺展好每一轮所需的密匙数组
3. 对被测目标 flag 的关键内文本进行二等分切割转送成两大 16 字节信息块（末端如过短则强行挂 0）
4. 放手对这两坨数据组分别进行魔方般的 `_f_enc` 不可逆转化推算
5. 反解开预置于文件死角的一段 Base64 硬编码串文本：

```text
s1iKn8YgOi8DO4p7bWHpTldoIwNL38IWGlcBbFngHRo=
```

6. 对这些 Base64 转录得来的值跟常数指引物 `0x66` 按字节级别循环上演整整 32 次的层叠互斥运算
7. 最后那道步骤自然是把魔方变化出来的双拼秘卷跟上述预置量做对等的比特级打听认证

弄清楚了它想怎么玩以后，最强有力的脱靶利刃当然绝不是靠爆内存撞树，反之：

> **直接将内置好的最终密文对换思路，依靠之前掌握在手里的那套自制篡权 S-box 顺带真秘钥进行解构，反查倒推出真实未包裹明文**

### Step 5: 依托脱机脚本实现无损旗舰解锁

针对刚才复盘好的三个中台大动作，下方提供的 Python 源本已经做到了完整的逻辑克隆和反推模拟：

1. 直连 `hint.png` 进行破片化截取自制版用表 S-box
2. 将 `q` 作为根脉计算换回原始 32 字节源密匙
3. 利用 Python 环境重组伪 AES-256 原理把锁住的盲盒完全反剥到文字形态明文上

```python
from io import BytesIO
from base64 import b64decode
from zipfile import ZipFile
from PIL import Image
import hashlib


APK_PATH = "iscc/附件/app-release.apk"
Q = "DDFGXVGFAVVD"
TARGET_B64 = "s1iKn8YgOi8DO4p7bWHpTldoIwNL38IWGlcBbFngHRo="
SQUARE = "mh7q1au6sl5pcv20t9zedwxfnjrgi8bk4y3o"
ROW_MAP = {"A": 0, "D": 6, "F": 12, "G": 18, "V": 24, "X": 30}
COL_MAP = {"A": 0, "D": 1, "F": 2, "G": 3, "V": 4, "X": 5}
RCON = [0x01, 0x02, 0x04, 0x08, 0x10, 0x20, 0x40, 0x80, 0x1B, 0x36]


def extract_sbox_from_apk(apk_path: str) -> bytes:
    with ZipFile(apk_path, "r") as zf:
        png_data = zf.read("assets/hint.png")

    img = Image.open(BytesIO(png_data)).convert("RGBA")
    w, h = img.size
    pixels = list(img.getdata())

    bits = []
    for y in range(h):
        for x in range(w):
            bits.append(pixels[y * w + x][2] & 1)
            if len(bits) >= 0x830:
                break
        if len(bits) >= 0x830:
            break

    raw = bytearray(0x830 // 8)
    for i, b in enumerate(bits):
        if b:
            raw[i >> 3] |= 1 << (7 - (i & 7))

    assert raw[:4] == b"MEOW"
    return bytes(raw[4:4 + 256])


def derive_key(q: str, sbox: bytes) -> bytes:
    perm = q[1::4] + q[0::4] + q[2::4] + q[3::4]
    out = []
    for i in range(32):
        a = perm[(2 * i) % len(perm)]
        b = perm[(2 * i + 1) % len(perm)]
        ch = SQUARE[ROW_MAP[a] + COL_MAP[b]]
        out.append(sbox[ord(ch)] ^ ((0x13 * i) & 0xFF))
    return bytes(out)


def sub_word(word: bytes, sbox: bytes) -> bytes:
    return bytes(sbox[b] for b in word)


def rot_word(word: bytes) -> bytes:
    return word[1:] + word[:1]


def expand_key(key: bytes, sbox: bytes) -> list[bytes]:
    words = [key[i:i + 4] for i in range(0, 32, 4)]
    for i in range(8, 60):
        temp = words[i - 1]
        if i % 8 == 0:
            temp = bytes(
                a ^ b
                for a, b in zip(
                    sub_word(rot_word(temp), sbox),
                    bytes([RCON[i // 8 - 1], 0, 0, 0]),
                )
            )
        elif i % 8 == 4:
            temp = sub_word(temp, sbox)
        words.append(bytes(a ^ b for a, b in zip(words[i - 8], temp)))
    return [b"".join(words[4 * r:4 * r + 4]) for r in range(15)]


def add_round_key(state: list[int], key: bytes) -> list[int]:
    return [a ^ b for a, b in zip(state, key)]


def gmul(a: int, b: int) -> int:
    p = 0
    for _ in range(8):
        if b & 1:
            p ^= a
        hi = a & 0x80
        a = (a << 1) & 0xFF
        if hi:
            a ^= 0x1B
        b >>= 1
    return p


def inv_shift_rows(state: list[int]) -> list[int]:
    rows = [state[i::4] for i in range(4)]
    out = [[0] * 4 for _ in range(4)]
    for r in range(4):
        row = list(rows[r])
        row = row[-r:] + row[:-r] if r else row
        out[r] = row
    return [out[r][c] for c in range(4) for r in range(4)]


def inv_mix_single_col(col: list[int]) -> list[int]:
    a0, a1, a2, a3 = col
    return [
        gmul(a0, 14) ^ gmul(a1, 11) ^ gmul(a2, 13) ^ gmul(a3, 9),
        gmul(a0, 9) ^ gmul(a1, 14) ^ gmul(a2, 11) ^ gmul(a3, 13),
        gmul(a0, 13) ^ gmul(a1, 9) ^ gmul(a2, 14) ^ gmul(a3, 11),
        gmul(a0, 11) ^ gmul(a1, 13) ^ gmul(a2, 9) ^ gmul(a3, 14),
    ]


def inv_mix_columns(state: list[int]) -> list[int]:
    out = []
    for c in range(4):
        out.extend(inv_mix_single_col(state[4 * c:4 * c + 4]))
    return out


def decrypt_block(block: bytes, round_keys: list[bytes], inv_sbox: list[int]) -> bytes:
    state = list(add_round_key(list(block), round_keys[14]))
    for rnd in range(13, 0, -1):
        state = inv_shift_rows(state)
        state = [inv_sbox[b] for b in state]
        state = add_round_key(state, round_keys[rnd])
        state = inv_mix_columns(state)
    state = inv_shift_rows(state)
    state = [inv_sbox[b] for b in state]
    state = add_round_key(state, round_keys[0])
    return bytes(state)


def main():
    assert hashlib.md5(Q.encode()).hexdigest() == "72184e58e765a1a579b201c3e15bc0ce"

    sbox = extract_sbox_from_apk(APK_PATH)
    inv_sbox = [0] * 256
    for i, v in enumerate(sbox):
        inv_sbox[v] = i

    key = derive_key(Q, sbox)
    round_keys = expand_key(key, sbox)

    target = bytes(b ^ 0x66 for b in b64decode(TARGET_B64))
    pt1 = decrypt_block(target[:16], round_keys, inv_sbox)
    pt2 = decrypt_block(target[16:], round_keys, inv_sbox)

    inner = (pt1 + pt2).rstrip(b"\x00").decode()
    flag = f"ISCC{{{inner}}}"

    print(f"[+] q       = {Q}")
    print(f"[+] key     = {key.hex()}")
    print(f"[+] block1  = {pt1!r}")
    print(f"[+] block2  = {pt2!r}")
    print("=" * 50)
    print(f"[!!!] FLAG: {flag}")
    print("=" * 50)


if __name__ == "__main__":
    main()
```
