
## 详细解题步骤

### Step 1: 注册与登录 (获取初始凭证)
题目提供了一个基础的用户系统，包含 `/register` 和 `/login` 接口。
* **动作**：首先随便注册一个普通用户（脚本中生成了随机用户名 `u` + 8位 Hex），并使用该账号密码登录。
* **目的**：登录成功后，从服务端的 Cookie 中获取初始的合法 `token`（JWT 格式）。

### Step 2: 伪造恢复模式 JWT (JWT 校验漏洞)
获取到正常的 JWT token 后，需要进行越权或状态切换。
* **动作**：将 JWT 按 `.` 分割为 Header、Payload、Signature 三部分。对 Payload 进行 Base64 解码，修改其中的关键字段：
  * 将 `"status"` 字段修改为 `"RECOVERY"`。
  * 将 `"exp"`（过期时间）向后延长 3600 秒。
* **重组**：将修改后的 Payload 重新 Base64 编码，**并直接拼接上原有的 Header 和 原有的 Signature**。
* **漏洞点**：这里是一个非常典型的 JWT 校验逻辑漏洞（脚本注释称为 `jwt_downgrade exploit`）。服务端在解析 JWT 时，可能只要看到 `status` 为 `RECOVERY`，就跳过了对 Signature（签名）的严格校验，或者底层的 JWT 库存在配置错误，导致攻击者无需知道 secret 即可篡改 payload。

### Step 3: 触发 SQL 报错注入 (探测注入点)
携带伪造好的 `RECOVERY` 状态的 Token（放入 Cookie 中），获得了访问敏感 API 的权限。
* **目标接口**：`POST /api/v1/user/sync`
* **传参方式**：JSON 格式。
* **动作**：向该接口发送包含单引号 `'` 的探测 Payload：`{"theme": "'", "language": "zh"}`。
* **现象**：服务端处理 `theme` 字段时没有进行参数绑定或过滤，直接拼接进了 SQL 语句中，导致数据库报错，并且**服务端将 SQL 报错信息直接返回给了前端**（响应体中泄露了错误信息）。

### Step 4: 利用 updatexml 进行报错注入取数据
由于可以回显报错信息，可以构造基于 `updatexml()` 的报错注入来拖取数据库中的信息。
* **构造 Payload**：
  ```sql
  A' OR updatexml(1,concat(0x7e,substring(({SQL语句}),{起始位置},{步长}),0x7e),1)-- 
  ```
  将其放入 JSON 的 `theme` 字段中：
  ```json
  {"theme": "A' OR updatexml(1,concat(0x7e,substring((SELECT flag FROM flags LIMIT 0,1),1,28),0x7e),1)-- ", "language": "zh"}
  ```
* **原理分析**：
  * `0x7e` 是波浪号 `~` 的 Hex 编码，用于在报错结果中作为边界符，方便用正则 `~(.*?)~` 提取真正的数据。
  * `updatexml()` 函数在第二个参数不是合法的 XPath 格式时会抛出 `XPATH syntax error`，并附带计算后的结果。
  * **分段读取限制**：由于 MySQL 的 `updatexml()` 报错回显最大长度限制为 32 个字符，完整的 Flag（通常大于 32 字符）无法一次性显示。因此脚本中使用了 `substring(expr, pos, chunk_size)`，以 28 个字符为步长（留出空间给前后的 `~`），循环递增 `pos` 变量，多次请求分段拼接出完整的 Flag。

### Step 5: 提取 Flag
* 脚本默认执行的 SQL 查询是 `SELECT flag FROM flags LIMIT 0,1`。
* 不断发送请求，提取报错中的 `XPATH syntax error: '~...~'` 内容。
* 将每次提取出来的字符串切片拼接在一起，最终在终端打印出完整的 Flag 字符串。
