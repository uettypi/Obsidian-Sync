# 周报（20260406-20260412）
## 1.2026ciscn&ccb分区赛 easy_time
本周我尝试做了一下学长们夺冠的2026ciscn&ccb分区赛中的一道Web题——easy_time，以下是我的Write Up。

1. 首先先审计index.py中关于login页面的代码，可知h2应为密码md5两次之后的结果，故利用cmd5这个网站即可得到密码应为secret。![image.png](https://img.uettypi.top/2026/04/20260406213214799.png)
2. 成功登录进入后台之后，有3个功能，粗略地看了一下，如果存在漏洞的话，大概是：插件上传——文件上传、留言板——SSTI/XSS、个人About/头像——SSRF&文件上传。然后，开始审计相关功能的代码。审完后，可以发现，所有用到模板的地方都是传参，没有拼接，可见并不存在SSTI；然后是SSRF，这个有，但是由于下面这段代码限制了协议的类型，所以各类伪协议都不能用了：
```python
if parsed.scheme not in {"http", "https"}:
	return None
```
3. 最后是文件上传，这里有两个与之相关的功能，一个是插件上传，另一个是头像上传。插件上传粗审感觉是压缩包软链接，但这个我不太熟悉，所以先审了头像上传，这个看起来是要上传一个图片马，但后缀名是白名单，且强制重命名，所以这里线索断了，只能审插件上传了。
4. 插件上传看不懂捏......
5. 做到这里的时候思路断了，于是尝试在本地利用题目附件中给出的Dokerfile构建了一个Docker，这下发现了一个重大的突破点：靶机的80端口存在一个PHP服务！![image.png](https://img.uettypi.top/2026/04/20260407130109666.png)（由Dockerfile可知，靶机暴露了两个端口，经过测试，5000端口为Python服务，即靶机开启后默认进入的服务，而80端口则是一个隐藏的PHP服务）![image.png](https://img.uettypi.top/2026/04/20260407130444727.png)
6. 接下来的思路就比较清晰了：先利用文件上传上传WebShell，再利用PHP服务将其解析即可。所以还是得审计插件上传的代码，这里就利用AI帮我审计了一下，可知确实存在文件上传漏洞，且为ZipSlip。![image.png](https://img.uettypi.top/2026/04/20260407133524659.png)
7. 知道了是ZipSlip之后，就好办了，这里直接找了个ZipSlip的Python脚本，然后先在evil.php中写入一句话木马，再修改软链接路径为PHP服务所在的目录，随后生成恶意压缩包，最后利用插件上传上传并解压WebShell即可。
```python
import zipfile

if __name__ == "__main__":
	try:
		zipFile = zipfile.ZipFile("poc.zip", "a", zipfile.ZIP_DEFLATED)
		info = zipfile.ZipInfo("poc.zip")
		zipFile.write("evil.php", "../../../../../var/www/html/evil.php",zipfile.ZIP_DEFLATED)
		zipFile.close()
	except IOError as e:
		raise e
```
8. 现在，WebShell已经成功上传并解压了，但是，本题中的PHP服务并不暴露在外部，因此，这里需要利用到上面第2步中发现的SSRF注入点。但当我满怀期待地输入`http://127.0.0.1:80/evil.php?cmd=system("cat%20/flag");`后，却发现，返回了滚木！我的flag呢？于是，马上进入Docker中一探究竟！最终，发现flag并不在/flag，而在/tmp/flag，故将SSRF注入语句中的/flag替换为/tmp/flag，即可得到flag。![image.png](https://img.uettypi.top/2026/04/20260408143149833.png)![image.png](https://img.uettypi.top/2026/04/20260408143258281.png)

## 2.2024强网杯（简化版） password_game (simplify)
校内平台上，这道题正好在上一题的旁边，就顺便做了。

1. 首先，这道题有一个非常非常坑的点，就是在其guest类的__tostring()魔术方法中，出现了一个`$value();`的写法，这是错的！正确的写法应该是`$this->value();`。我一开始并没有注意到这一点，一直在尝试使用`user::__destruct()->guest::__tostring()->user::__invoke()->guest::__call($key,$value)`这一利用链。但这显然是做不到的，因为无论我如何尝试，都会报错`Fatal error: Method guest::__toString() must not throw an exception, caught Error: Function name must be a string in /var/www/html/index.php on line 54`，所以我不得不放弃这条路径。
2. 但是，如果要使用包含root类的路径的话，就必须得先想办法绕过基于preg_replace()的黑名单校验，这里我确实不太会，所以就在网上找了[这道题的WP](https://redshome.top/posts/2024-11-05-2024-%E5%BC%BA%E7%BD%91%E6%9D%AF-password-game-writeup/)，并且从中学到了两个比较细节的知识点：1.将序列化后的payload中的小写s改为大写S后，即可在字符串中使用16进制，用以绕过preg_replace()的正则匹配；2.PHP中&表示引用。（但是这里我必须要插一句，源WP中“PHP中的&其实就是和C语言里差不多一样的指针”的说法，其实是不对的，引用和指针是不同的两个概念，不能混为一谈。）
3. 到这里为止，这道题其实已经解决了，但是我还发现了一个（对我而言）比较抽象的点，就是，源WP中提供的payload，是将“2024qwb”中的“4”替换为16进制的“\34”的，然后我就在想，如果将其他字符替换掉的话，按理来说也应该是一样的，但结果是，当我尝试将第一个“2”替换为它的16进制“\32”时，却不行了！没想到这个问题竟然困扰了我接近半个小时，最终破案了：“\32”中的“2”和原本后面的“024qwb”又重新拼接成了完整的“2024qwb”，触发了正则匹配......
